## Глоссарий-минимум: что нужно понимать до старта

Эти понятия повторяются почти во всех ответах. Прочитайте один раз — дальше будет легче.

**AWS-аккаунт (account).** Базовая единица изоляции в AWS. Это не «логин пользователя», а целый контейнер ресурсов со своим биллингом и границей безопасности. Крупные компании держат не один аккаунт, а десятки и сотни — например, отдельный аккаунт под прод, под разработку, под логи, под безопасность. Так радиус поражения при компрометации одного аккаунта ограничен.

**Регион (Region) и зона доступности (Availability Zone, AZ).** Регион — это географическая область (например, `eu-central-1` во Франкфурте). Внутри региона есть несколько изолированных дата-центров — зон доступности. Размещая ресурсы в нескольких AZ, вы переживаете отказ одного дата-центра.

**IAM (Identity and Access Management).** Сервис управления тем, *кто* и *что* может делать в аккаунте. Ключевые объекты: пользователи, роли и политики.
- **Политика (policy)** — JSON-документ с правилами «разрешить/запретить действие над ресурсом». Например, «разрешить читать из конкретного S3-бакета».
- **Роль (role)** — набор прав, который *временно* на себя «надевают» (assume) пользователи, сервисы или приложения. У роли нет постоянного пароля; доступ выдаётся через временные учётные данные. Это основной способ безопасной выдачи прав в AWS.
- **Принцип наименьших привилегий (least privilege)** — выдавать ровно столько прав, сколько нужно для задачи, и не больше.

**S3 (Simple Storage Service).** Объектное хранилище — «бесконечный диск» для файлов (объектов), сгруппированных в бакеты (buckets). Используется для данных, бэкапов, логов, статики сайтов.

**VPC (Virtual Private Cloud).** Ваша изолированная виртуальная сеть внутри AWS. В ней вы создаёте подсети (subnets), задаёте маршрутизацию, фаерволы. Всё «сетевое» живёт внутри VPC.

**EC2 (Elastic Compute Cloud).** Виртуальные машины (инстансы) в облаке. Классический способ «запустить сервер».

**Lambda.** «Бессерверные» (serverless) функции: вы загружаете код, AWS сам его запускает по событию и масштабирует. Вы не управляете серверами вообще — отсюда «serverless».

**CloudTrail.** Журнал всех управляющих действий в аккаунте: кто, когда, какой API-вызов сделал (например, «пользователь X удалил бакет Y»). Это основа аудита и расследований.

**CloudWatch.** Сервис мониторинга: собирает метрики (числовые показатели), логи приложений и позволяет настраивать алармы (alarms) — автоматические оповещения при выходе показателя за порог.

**KMS (Key Management Service).** Управляемое хранилище и сервис криптографических ключей. С его помощью шифруют данные «в покое» (at rest) практически во всех сервисах AWS, не храня сами ключи в коде.

**AWS Organizations.** Сервис для управления множеством аккаунтов как единой структурой: иерархия, общий биллинг и централизованные политики-ограничения.

**SCP (Service Control Policy).** Политика уровня организации, которая задаёт *максимально возможные* права для аккаунтов (граница), даже для администраторов. SCP ничего не «разрешает» сама по себе — она ограничивает, что вообще можно делать в аккаунте.

**EventBridge.** Шина событий: принимает события от сервисов AWS (например, «GuardDuty нашёл угрозу») и по правилам направляет их в нужные обработчики (Lambda, SNS, очередь и т.д.). Это «нервная система» для автоматизации.

**SNS (Simple Notification Service).** Сервис уведомлений по схеме «публикация/подписка»: одно событие рассылается всем подписчикам (email, SMS, очередь, Lambda). Часто используется для алертов.

**IaC (Infrastructure as Code).** Подход, при котором инфраструктура описывается кодом (шаблонами), а не настраивается руками в консоли. В AWS это в первую очередь CloudFormation и Terraform. Даёт повторяемость, ревью изменений и аудит.

Дальше каждый специфичный сервис (GuardDuty, Security Hub, WAF, Config и т.д.) объясняется в том месте, где впервые встречается.

## Раздел 1. Мониторинг, логирование и оповещение

### 1.1. Мониторинг и оповещение в многоаккаунтной среде

**Задача.** Спроектировать и внедрить надёжное решение для мониторинга и оповещений в среде из множества AWS-аккаунтов: проанализировать рабочие нагрузки, выбрать стратегию мониторинга, агрегировать события безопасности, построить полезные метрики и дашборды, автоматизировать постоянную оценку защищённости. Назвать конкретные сервисы, описать риски и обеспечить масштабируемость по мере роста.

#### Шаг 0. Сначала понять, что мы вообще защищаем

Прежде чем включать сервисы, проводят **инвентаризацию и анализ рабочих нагрузок**. Нельзя осмысленно мониторить то, чего не знаешь. На практике это значит ответить на вопросы:

- Какие аккаунты есть и за что отвечает каждый (прод, dev, песочницы, аккаунт безопасности, аккаунт логов)?
- Где лежат критичные данные (база с клиентами, S3 с документами) и какие сервисы их трогают?
- Что является «короной» (crown jewels) — компрометация чего нанесёт максимальный ущерб?
- Какие требования по соответствию применимы (PCI DSS для платёжных данных, HIPAA для медицинских и т.д.)?

Результат — карта активов и модель угроз. Из неё рождается список того, *за чем именно* следить.

> **Новый термин — модель угроз (threat model).** Структурированный ответ на вопросы «что мы защищаем», «от кого», «какими путями нас могут атаковать» и «что мы будем делать». Без неё мониторинг превращается в случайный набор алертов.

#### Шаг 1. Базовый фундамент данных: трейлы и потоки

Любой мониторинг строится на источниках данных. Минимально необходимый набор:

**AWS CloudTrail — журнал API-вызовов.** Записывает управляющие действия во всех аккаунтах. В многоаккаунтной среде создаётся **organization trail** — один общий трейл на уровне AWS Organizations, который автоматически собирает события со *всех* аккаунтов (включая будущие, которые ещё не созданы) в один центральный S3-бакет. Это критично: иначе каждый аккаунт логировал бы себя сам, и часть команд просто забыла бы включить трейл.

**Amazon VPC Flow Logs — журнал сетевого трафика.** Записывает метаданные о соединениях внутри VPC (кто с кем, по какому порту, разрешено или отброшено). Нужны для расследования сетевых аномалий — например, инстанс внезапно стучится на странный внешний IP.

**Логи приложений и сервисов в CloudWatch Logs.** Lambda, контейнеры, нагрузочные балансировщики, базы данных пишут логи сюда.

> **Принцип:** данные должны собираться централизованно и быть защищены от изменения. Логи безопасности складывают в **отдельный, выделенный аккаунт «Log Archive»**, доступ к которому максимально ограничен. Идея проста: если злоумышленник скомпрометировал прод-аккаунт, он не должен иметь возможности удалить логи, которые его выдают. Бакет с логами защищают неизменяемостью (об этом — в разделе про S3 Object Lock, 5.2).

#### Шаг 2. Слой обнаружения угроз

Сырые логи — это много данных, но мало смысла. Поверх них ставят сервисы, которые сами находят подозрительное.

**Amazon GuardDuty — управляемое обнаружение угроз.** Это сервис, который непрерывно анализирует CloudTrail, VPC Flow Logs, DNS-запросы (а опционально — события из S3, EKS, RDS, Lambda) и с помощью машинного обучения и баз индикаторов угроз выдаёт находки (findings): «инстанс общается с известным C2-сервером», «необычный вызов API из нетипичной страны», «попытка перебора SSH». Вам не нужно писать правила — GuardDuty уже знает типовые атаки. В организации его включают через **делегированного администратора** (об этом ниже) сразу на все аккаунты.

**Amazon Inspector — поиск уязвимостей.** Автоматически сканирует EC2-инстансы, образы контейнеров в ECR и Lambda-функции на известные уязвимости (CVE) и проблемы конфигурации. Отвечает на вопрос «где у нас дырявый софт».

**AWS Config — отслеживание конфигураций и соответствия.** Записывает *состояние* ресурсов во времени («у этого бакета вчера было выключено шифрование») и проверяет их на соответствие правилам (Config Rules). Например, правило «все тома EBS должны быть зашифрованы» — Config помечает нарушителей. Это «постоянная оценка защищённости», о которой спрашивает задача.

> **Новый термин — делегированный администратор (delegated administrator).** В AWS Organizations большинство security-сервисов (GuardDuty, Security Hub, Config, Inspector, Detective) умеют назначать один аккаунт «главным» по этому сервису для всей организации. Обычно это выделенный **Security/Audit-аккаунт**. Через него вы централизованно видите находки со всех аккаунтов и управляете настройками, не заходя в каждый.

#### Шаг 3. Агрегация: единое окно вместо десятка консолей

Если в каждом аккаунте свои находки GuardDuty, Inspector и Config — инженер утонет. Нужна **агрегация**.

**AWS Security Hub — центральный агрегатор находок.** Собирает находки из GuardDuty, Inspector, Config, Macie и сторонних инструментов в *едином формате* (ASFF — AWS Security Finding Format) и в одном месте. Дополнительно Security Hub прогоняет вашу среду по готовым наборам стандартов (CIS AWS Foundations Benchmark, AWS Foundational Security Best Practices, PCI DSS) и показывает оценку соответствия в процентах. В организации назначается делегированный администратор Security Hub, и все находки стекаются в Security-аккаунт.

> **Новый термин — CIS Benchmark / Foundational Security Best Practices.** Это опубликованные наборы проверяемых правил «как должна быть настроена безопасная среда AWS» (например, «MFA включён на root», «CloudTrail включён во всех регионах»). Security Hub автоматически проверяет вашу среду против них.

**Amazon Detective (опционально) — расследование.** Когда находка получена, Detective строит граф связей (какой пользователь, с какого IP, какие ресурсы трогал) и помогает быстро понять масштаб. Это ускоряет триаж.

#### Шаг 4. Метрики, дашборды и «сигнал против шума»

Главная ошибка мониторинга — заваливать команду алертами, на которые перестают реагировать (alert fatigue). Поэтому:

- **Метрики строят вокруг рисков, а не вокруг всего подряд.** Полезные метрики: число критичных находок, не закрытых дольше N часов; процент аккаунтов с включённым GuardDuty; число попыток доступа к root; доля ресурсов, нарушающих Config-правила; время от находки до реакции (MTTR).
- **Дашборды** делают в **Amazon CloudWatch Dashboards** (для операционных метрик) и в **Security Hub summary** (для security-постуры). Для сложной аналитики и красивых отчётов подключают **Amazon QuickSight** (BI-инструмент AWS) поверх данных, выгруженных в S3.
- **Приоритизация алертов.** Не каждая находка должна будить человека ночью. Критичные (активная компрометация) → немедленный пейджер; средние → тикет в систему задач; информационные → просто копятся для отчётов.

#### Шаг 5. Автоматизация: от находки к действию без человека

Это сердце масштабируемого решения. Схема стандартная и встречается во всех разделах этого документа:

```
Источник события (GuardDuty / Security Hub / Config)
        │  событие
        ▼
   Amazon EventBridge  ──правило фильтрации──►  обработчик
        │                                         │
        │                                         ├──► SNS  → email / Slack / PagerDuty (оповестить людей)
        │                                         ├──► Lambda → автоматическое исправление (remediation)
        │                                         └──► Step Functions → сложный многошаговый сценарий
```

> **Новый термин — EventBridge.** Шина событий. Security-сервисы публикуют в неё события автоматически. Вы пишете **правило** («если пришла критичная находка GuardDuty типа X») и указываете **цель** (Lambda, SNS, очередь). Так строится реакция в реальном времени.

> **Новый термин — remediation (автоустранение).** Lambda-функция, которая при определённом событии сама чинит проблему: закрывает публичный доступ к бакету, изолирует скомпрометированный инстанс, отзывает ключ. Для сложных цепочек используют **AWS Step Functions** — сервис оркестрации, где шаги выстроены в визуальный граф с условиями и повторами.

**AWS Systems Manager (SSM)** часто служит «руками» автоматизации: через него выполняют команды на инстансах, применяют патчи, запускают runbook-и (готовые сценарии действий).

#### Конкретный стек сервисов (резюме)

| Назначение | Сервис |
|---|---|
| Журнал API-действий | CloudTrail (organization trail) |
| Сетевые логи | VPC Flow Logs |
| Логи приложений | CloudWatch Logs |
| Обнаружение угроз | Amazon GuardDuty |
| Поиск уязвимостей | Amazon Inspector |
| Конфигурации и соответствие | AWS Config |
| Агрегация находок | AWS Security Hub |
| Расследование | Amazon Detective |
| Шина событий | Amazon EventBridge |
| Оповещения | Amazon SNS |
| Автоустранение | Lambda, Step Functions, SSM |
| Дашборды | CloudWatch Dashboards, QuickSight |
| Хранилище логов | S3 в выделенном Log-аккаунте |

#### Потенциальные проблемы и как их решать

- **Стоимость.** GuardDuty, Config и Security Hub тарифицируются по объёму обрабатываемых событий. На больших объёмах это заметные деньги. Решение: фильтровать «болтливые» источники, складывать сырые логи в дешёвые классы хранения S3, использовать выборку там, где уместно.
- **Шум алертов.** Лечится приоритизацией, подавлением известных ложных срабатываний (suppression rules в Security Hub) и автоматическим закрытием неважных находок.
- **Дрейф конфигураций при росте.** Новые команды создают аккаунты и забывают включить мониторинг. Решение: включать сервисы на уровне организации (auto-enable для новых аккаунтов) и форсировать базовые настройки через SCP и Control Tower (раздел 6).
- **Защита самих логов.** Если злоумышленник может стереть логи — мониторинг бесполезен. Логи изолируют в отдельном аккаунте с неизменяемостью.

#### Как обеспечивается масштабируемость

Ключевая идея: **всё включается на уровне организации, а не аккаунта.** Делегированные администраторы security-сервисов + автоподключение новых аккаунтов + organization trail означают, что при создании 10-го или 500-го аккаунта мониторинг применяется автоматически, без ручной работы. Сам пайплайн (EventBridge → Lambda/SNS) событийный и горизонтально масштабируемый по природе. Инфраструктуру мониторинга разворачивают через IaC (CloudFormation StackSets — об этом в 6.2), чтобы она была воспроизводимой и проверяемой.

### 1.2. Стратегия логирования для крупного предприятия

**Задача.** Разработать комплексную стратегию логирования для большого предприятия со сложной сетью (много VPC, транзитные шлюзы, гибридные подключения). Нужно: выявить источники логов, продумать приём/хранение/срок хранения, анализ и корреляцию, интеграцию со сторонними инструментами, масштабируемость и оптимизацию затрат, соответствие требованиям и мониторинг в реальном времени.

> **Новый термин — Transit Gateway (транзитный шлюз).** Представьте сетевой «хаб», к которому подключаются все ваши VPC и каналы в свой дата-центр. Вместо запутанной паутины соединений «каждый с каждым» всё идёт через один шлюз. Это упрощает маршрутизацию, но делает шлюз критичной точкой для логирования сетевого трафика.

> **Новый термин — гибридное подключение (hybrid connectivity).** Связь между AWS и вашим собственным дата-центром («он-прем»). Реализуется через VPN или через **AWS Direct Connect** (выделенный физический канал). Логи на стыке облака и он-према особенно важны для расследований.

#### Шаг 1. Идентификация источников логов

Полная стратегия охватывает несколько слоёв. Для каждого слоя — свой источник:

| Слой | Источник логов | Что фиксирует |
|---|---|---|
| Управление аккаунтом | CloudTrail (management + data events) | API-вызовы, доступ к данным в S3/DynamoDB |
| Сеть (внутри VPC) | VPC Flow Logs | соединения, разрешённые/отброшенные |
| Сеть (DNS) | Route 53 Resolver Query Logs | DNS-запросы (часто выдают malware) |
| Транзитный шлюз | Transit Gateway Flow Logs | трафик между VPC и он-премом |
| Веб-периметр | WAF Logs, CloudFront Logs, ALB Access Logs | HTTP-запросы, атаки на приложения |
| Приложения | CloudWatch Logs / агенты | бизнес-логика, ошибки |
| Базы данных | RDS/Aurora audit logs, DynamoDB Streams | доступ к данным, изменения |
| Угрозы | GuardDuty findings | готовые выводы об атаках |

> **Новый термин — data events в CloudTrail.** По умолчанию CloudTrail пишет только *управляющие* действия (создал/удалил ресурс). Data events дополнительно фиксируют *доступ к данным* — например, конкретное чтение объекта из S3 или запись в DynamoDB. Они объёмны и платны, поэтому включаются точечно для критичных бакетов и таблиц, но для расследований и соответствия часто обязательны.

> **Новый термин — AWS WAF (Web Application Firewall).** Фаервол уровня приложения: фильтрует HTTP-запросы по правилам (блокировать SQL-инъекции, XSS, заходы с плохих IP, превышение лимита запросов). Ставится перед CloudFront, ALB или API Gateway. Его логи — ценный источник для анализа веб-атак.

#### Шаг 2. Приём, хранение и срок хранения (ingestion → storage → retention)

**Централизованный сбор.** Все логи стекаются в выделенный **Log Archive аккаунт**, в S3. CloudTrail-organization-trail и Config пишут туда напрямую; логи из CloudWatch Logs экспортируются (через подписки или Firehose); сетевые логи направляются в тот же бакет.

> **Новый термин — Amazon Kinesis Data Firehose.** Сервис потоковой доставки данных: принимает поток событий и буферизует/сжимает/доставляет их в S3, OpenSearch или сторонние SIEM почти в реальном времени. Часто стоит «посередине» между источником логов и хранилищем/аналитикой.

**Многоуровневое хранение для оптимизации затрат.** Логи — это «горячо сейчас, холодно потом». Поэтому используют классы хранения S3 и переходы по жизненному циклу:

- **0–90 дней (горячие):** S3 Standard + индексация в системе анализа — нужны для оперативных расследований.
- **90 дней – 1 год (тёплые):** S3 Standard-IA / Intelligent-Tiering — реже запрашиваются.
- **1–7+ лет (холодные/архив):** S3 Glacier / Glacier Deep Archive — дёшево, для соответствия и редких форензик-запросов.

> **Новый термин — S3 Lifecycle policy.** Правило автоматического перемещения объектов между классами хранения по возрасту и удаления по истечении срока. Это главный инструмент экономии на логах: вы не платите за «горячий» класс для трёхлетних логов.

**Срок хранения (retention)** задаётся требованиями: например, PCI DSS требует хранить определённые логи не менее года (с доступом к недавним за 3 месяца), некоторые финансовые/медицинские нормы — 7–10 лет. Для неизменяемости применяют **S3 Object Lock** (WORM — write once, read many): объект нельзя удалить или изменить до окончания срока, даже администратору. Это защищает логи от удаления злоумышленником и удовлетворяет аудиторов.

#### Шаг 3. Анализ и корреляция

Хранить логи мало — нужно уметь искать в них и связывать события.

**Amazon Athena.** Сервис, позволяющий выполнять SQL-запросы прямо по файлам в S3, без загрузки в базу. Идеально для разовых расследований: «покажи все вызовы DeleteObject в этом бакете за вчера». Дёшев (платите за просканированные данные), особенно если логи в колоночном формате (Parquet) и партиционированы по дате.

**Amazon OpenSearch Service.** Поисково-аналитический движок (форк Elasticsearch) с дашбордами. Хорош для интерактивного просмотра логов в реальном времени и сложной корреляции. Дороже Athena в постоянной работе, поэтому в нём держат «горячие» логи, а холод оставляют в S3 под Athena.

**Корреляция событий** — это связывание разрозненных логов в единую картину атаки. Пример: GuardDuty сообщает о подозрительном API-вызове → по CloudTrail находим, какая роль его сделала → по VPC Flow Logs видим, куда инстанс ходил по сети → по WAF-логам — каким запросом его пробили. Корреляцию делают либо в SIEM (см. ниже), либо в OpenSearch, либо автоматически в Amazon Detective.

> **Новый термин — SIEM (Security Information and Event Management).** Класс систем, которые собирают логи из множества источников, нормализуют их, коррелируют по правилам и выдают приоритизированные алерты. Это «командный центр» SOC. В AWS роль SIEM может играть OpenSearch с готовым решением, либо сторонний продукт (Splunk, Microsoft Sentinel, Elastic).

> **Новый термин — SOC (Security Operations Center).** Команда (и/или платформа), которая круглосуточно следит за событиями безопасности, расследует и реагирует.

#### Шаг 4. Интеграция со сторонними инструментами

Многие предприятия уже используют Splunk, Datadog, Sumo Logic, Chronicle и т.п. AWS интегрируется с ними так:

- **Через Kinesis Data Firehose** — поток логов доставляется прямо в сторонний приёмник.
- **Через Security Hub** — он экспортирует находки в общем формате ASFF, которые сторонние SIEM умеют забирать (а партнёрские продукты умеют и присылать находки *в* Security Hub).
- **Через Amazon Security Lake** — относительно новый сервис, который автоматически централизует логи безопасности из AWS и сторонних источников в озеро данных в *открытом нормализованном формате OCSF*. Это устраняет ручную возню с разными форматами и даёт сторонним инструментам единый источник.

> **Новый термин — OCSF (Open Cybersecurity Schema Framework).** Открытый стандарт описания событий безопасности в едином виде. Когда логи приведены к OCSF, любой совместимый инструмент анализирует их без кастомных парсеров. Security Lake складывает данные именно в OCSF.

#### Шаг 5. Масштабируемость и оптимизация затрат

- **Колоночные форматы и партиционирование.** Хранение логов в Parquet с разбиением по дате/аккаунту резко удешевляет запросы Athena.
- **Lifecycle + Glacier** для старых данных (см. шаг 2).
- **Разделение «горячего» и «холодного».** Дорогой OpenSearch — только для недавних логов; всё остальное — дешёвый S3 + Athena.
- **Фильтрация на источнике.** Не всё нужно хранить вечно: отсекают заведомо бесполезный шум до доставки.
- **Security Lake** сам управляет жизненным циклом и форматами, снижая операционные расходы при росте.

#### Шаг 6. Соответствие требованиям

- Неизменяемость через **S3 Object Lock** + ограниченный доступ к Log-аккаунту = доказуемая целостность логов.
- Шифрование логов в покое через **KMS**, в движении — через TLS.
- Полный аудиторский след: кто читал/трогал логи — это тоже логируется (CloudTrail data events на бакет логов).
- Документированные сроки хранения под каждый стандарт (PCI/HIPAA/GDPR).

#### Шаг 7. Мониторинг и оповещение в реальном времени

- **CloudWatch Logs metric filters + alarms** — например, аларм при появлении в логах строки «root login» или N подряд ошибок авторизации.
- **EventBridge** — мгновенная реакция на находки GuardDuty/Security Hub.
- **CloudWatch Logs subscription filters → Lambda/Firehose** — потоковая обработка логов на лету (поймать аномалию, не дожидаясь батч-анализа).

#### Типичные проблемы

- **Взрыв объёмов и затрат** — лечится lifecycle, фильтрацией, Parquet, разделением горячее/холодное.
- **Разнородные форматы** — лечится нормализацией (Security Lake/OCSF).
- **Пробелы в покрытии** — какой-то VPC/аккаунт без Flow Logs. Лечится принудительным включением через организацию и проверками Config.
- **Слепые зоны на гибридном стыке** — обязательно логировать Transit Gateway и трафик через Direct Connect/VPN.

### 1.3. Диагностика проблем логирования и мониторинга

**Задача.** В организации хаос: часть логов отсутствует, алерты приходят с задержкой, мониторинг непоследователен в Lambda, API Gateway, CloudFront и кастомных приложениях. Нужно: систематически расследовать, проанализировать права и конфигурации, внедрить исправления, выстроить превентивные меры и устойчивый мониторинг. Описать методику, инструменты, частые ошибки, документацию/коммуникацию и долгосрочные улучшения.

Это типичная «пожарная» ситуация. Подходить к ней нужно как инженер, а не как угадывающий — по методике.

#### Методика устранения неполадок (общий каркас)

1. **Зафиксировать симптомы конкретно.** «Логов нет» — где именно, с какого времени, во всех аккаунтах или одном, во всех регионах или одном? Без конкретики чинить нельзя.
2. **Построить карту потока данных** для каждого затронутого сервиса: где лог *рождается* → куда *должен* доставляться → где *теряется*. Проблема всегда на одном из звеньев.
3. **Проверять от источника к приёмнику.** Сначала — пишется ли лог вообще; потом — есть ли права на доставку; потом — доходит ли до хранилища; потом — срабатывает ли алерт.
4. **Изолировать переменную.** Менять по одному, проверять эффект.
5. **Зафиксировать root cause и внедрить превентивную меру**, чтобы не повторилось.

#### Разбор по сервисам: где обычно теряются логи

**Lambda.**
- *Причина №1 — права.* Функция пишет логи в CloudWatch Logs через свою **execution role** (роль исполнения). Если у роли нет прав `logs:CreateLogGroup`/`logs:CreateLogStream`/`logs:PutLogEvents` — логов не будет, при этом сама функция работает. Это самая частая причина «пропавших» логов Lambda.
- *Причина №2 — retention/удаление.* Лог-группа могла иметь нулевой/короткий срок хранения, и логи просто истекли.
- *Решение:* добавить права в роль, задать политику retention, стандартизировать это в шаблоне развёртывания всех функций.

**API Gateway.**
- *Причина — логирование не включено.* В API Gateway есть два уровня логов: **access logs** (кто и что запросил) и **execution logs** (детали обработки). Оба надо включать явно на уровне stage, плюс настроить аккаунт-уровневую роль для записи в CloudWatch. По умолчанию многое выключено.
- *Решение:* включить access/execution logs, задать формат, проверить роль.

**CloudFront.**
- *Причина №1 — стандартные логи не включены* или пишутся в бакет, до которого нет доступа.
- *Причина №2 — задержка.* Стандартные (legacy) логи CloudFront доставляются в S3 *с задержкой в десятки минут* — это нормальная их природа, а не баг. Если нужны логи почти в реальном времени, используют **real-time logs** (через Kinesis) или новые логи через стандартный механизм доставки.
- *Решение:* включить логи, а для оперативного мониторинга — real-time logs; понимать, что часть «задержек алертов» — это просто латентность legacy-логов.

**Кастомные приложения (EC2/контейнеры).**
- *Причина — нет агента или он не настроен.* Чтобы логи приложения с инстанса попали в CloudWatch, нужен **CloudWatch agent** (или для контейнеров — драйвер логирования awslogs/Fluent Bit). Часто его забывают поставить или у инстанса нет роли с правами на отправку.
- *Решение:* стандартизировать установку агента через образ/Launch Template/SSM, выдать инстансам корректную роль.

#### Почему задержка алертов

Типичные причины и лечение:

- **Латентность источника** (как у CloudFront legacy-логов) → перейти на real-time logs там, где задержка критична.
- **Батч-обработка вместо потоковой** → заменить периодический разбор на subscription filters / EventBridge.
- **Метрика считается на длинном окне** → у CloudWatch-аларма слишком большой период оценки; уменьшить.
- **Перегруженный пайплайн** (Firehose/Lambda не успевают) → увеличить конкуррентность/буферы, проверить throttling.
- **Цепочка SNS → почта** медленная или письма уходят в спам → перейти на нормальный канал инцидентов (PagerDuty/Opsgenie/Slack).

#### Анализ прав и конфигураций

Корень большинства проблем — **IAM-права** и **ресурсные политики**. Что проверяют:

- У ролей исполнения (Lambda, инстансы, задачи ECS) — есть ли права писать логи и доставлять их.
- **Resource policy** приёмников: у целевого S3-бакета или лог-группы — разрешена ли запись от нужного сервиса (например, у бакета должна быть политика, позволяющая CloudTrail/CloudFront писать в него).
- **KMS-права:** если приёмник зашифрован ключом KMS, у пишущего сервиса должно быть право использовать этот ключ — иначе запись молча падает.
- **SCP на уровне организации:** не запрещает ли какая-то широкая SCP нужное действие. Бывает, что глобальный «запрет» рубит легитимное логирование.
- Инструменты: **IAM Access Analyzer** (находит публичный/некорректный доступ и неиспользуемые права), симулятор политик, и **CloudTrail** — чтобы увидеть `AccessDenied` по конкретному вызову.

> **Новый термин — IAM Access Analyzer.** Сервис, который анализирует политики и говорит, к каким ресурсам открыт доступ извне аккаунта/организации, а также (в режиме unused access) подсвечивает выданные, но не используемые права. Помогает и чинить доступ, и соблюдать least privilege.

#### Внедрение решений

1. Восстановить недостающие права (роли, ресурсные политики, KMS-доступы).
2. Включить недостающее логирование (API Gateway stages, CloudFront, data events).
3. Установить/починить агенты на инстансах и контейнерах.
4. Заменить медленные звенья на потоковые (real-time logs, subscription filters).
5. Стандартизировать сроки хранения лог-групп.

#### Превентивные меры (чтобы не повторилось)

- **Всё через IaC.** Логирование, роли и retention описываются в шаблонах (CloudFormation/Terraform). Тогда новый сервис разворачивается уже с правильным логированием.
- **AWS Config Rules как «детекторы дрейфа».** Правила вроде «у каждой Lambda должна быть лог-группа», «у API Gateway включены access logs», «CloudTrail включён» — Config будет ловить отклонения автоматически.
- **Guardrails через организацию** (Control Tower / SCP) — запрет выключать CloudTrail и т.п.
- **«Canary»-проверки.** Синтетический тест, который регулярно генерирует событие и проверяет, дошёл ли алерт за ожидаемое время — мониторинг самого мониторинга.

#### Устойчивая стратегия мониторинга (мониторим сам мониторинг)

Ключевая зрелая идея: **наблюдаемость должна сама себя проверять.** Метрики покрытия: «процент Lambda с рабочими лог-группами», «число аккаунтов без CloudTrail», «свежесть последнего полученного лога по каждому источнику» (если источник «замолчал» — алерт). Это превращает разовый разбор пожара в постоянный контроль.

#### Частые ошибки и как избежать

- **Чинить симптом, а не причину** (вручную включил логи в одном месте, а корень — отсутствующий стандарт). Лечение: IaC + Config Rules.
- **Забыть про KMS-права** при шифрованных приёмниках. Лечение: чек-лист «право на ключ» в ревью.
- **Считать задержку багом, когда это природа сервиса** (CloudFront legacy logs). Лечение: знать характеристики источников.
- **Слишком широкие SCP**, рубящие легитимное логирование. Лечение: тестировать политики в симуляторе.
- **Отсутствие проверки покрытия** — никто не замечает, что источник замолчал. Лечение: canary/freshness-алерты.

#### Документация и коммуникация

- Вести **runbook** (пошаговую инструкцию) по каждому классу проблемы — чтобы следующий инженер решал за минуты.
- Фиксировать инцидент и его root cause в **постмортеме** (разбор без поиска виноватых, с фокусом на исправление процесса).
- Поддерживать **карту источников логов** (что куда течёт) как живой документ.
- По ходу — прозрачно сообщать затронутым командам статус: что сломано, что чиним, когда восстановим.

#### Долгосрочные рекомендации

- Перейти к **централизованной наблюдаемости** (Log Archive аккаунт, Security Lake, единый SIEM).
- Внедрить **«мониторинг как код»**: дашборды, алармы и правила в репозитории под ревью.
- Регулярно проводить **game days** — учения, где намеренно ломают звено и проверяют, что система это замечает.

## Раздел 2. Реагирование на инциденты

> **Что такое реагирование на инциденты (Incident Response, IR).** Это дисциплина и процесс действий при инциденте безопасности. Классическая модель (NIST) состоит из стадий: **подготовка → обнаружение и анализ → локализация (containment) → устранение (eradication) → восстановление (recovery) → выводы (lessons learned)**. Хорошая IR — это не геройство во время пожара, а заранее отрепетированный план. Ниже эта модель ложится на сервисы AWS.

### 2.1. Разработка и тестирование плана реагирования на инциденты

**Задача.** Крупная компания с критичными нагрузками в нескольких AWS-аккаунтах, конфиденциальными данными клиентов и требованием высокой доступности. Нужно: разработать, внедрить и протестировать комплексный IR-план — ключевые компоненты и runbook-и, интеграция сервисов AWS для автоматизации, роли и обязанности, коммуникации; подготовка среды (инструменты, доступ, минимизация blast radius, guardrails); тестирование и валидация; автоматизированное устранение. Указать сервисы, проблемы и решения, метрики, непрерывное улучшение, соответствие.

#### Часть A. Ключевые компоненты плана и runbook-и

**IR-план** — это документ, отвечающий на вопросы «что считать инцидентом», «кто что делает», «как эскалируем», «как общаемся», «как восстанавливаемся». Внутри него — **runbook-и (playbooks)**: пошаговые инструкции под конкретные типы инцидентов.

Типовые playbook-и, которые стоит иметь:
- **Компрометация учётных данных / IAM** (утёк ключ доступа, подозрительный assume role).
- **Компрометация EC2-инстанса** (майнер, C2-коммуникация).
- **Публичная утечка данных** (бакет S3 открылся наружу).
- **Компрометация в Kubernetes/ECS** (подозрительный pod/контейнер).
- **DDoS / атака на веб-периметр**.
- **Программа-вымогатель** (массовое шифрование/удаление).

Каждый runbook описывает: как подтвердить инцидент, как локализовать, как собрать улики, как устранить, как восстановить, кого уведомить.

> **Новый термин — blast radius (радиус поражения).** Насколько далеко распространится ущерб, если конкретный компонент скомпрометирован. Цель архитектора — делать его минимальным: изоляция аккаунтов, наименьшие привилегии, сегментация сети. Тогда взлом одного сервиса не открывает всю организацию.

#### Часть B. Интеграция сервисов AWS для автоматизации

Связка та же, что в разделе 1, но повёрнутая на реакцию:

```
Обнаружение:   GuardDuty / Security Hub / Config / CloudWatch alarms
      │
      ▼
Маршрутизация: EventBridge (правила по типу и серьёзности находки)
      │
      ├──► SNS → оповещение дежурных (PagerDuty/Slack/email)
      ├──► Lambda → немедленная локализация (изолировать инстанс, отозвать ключ)
      └──► Step Functions → многошаговый playbook со сбором улик и эскалацией
```

Конкретные роли сервисов в IR:
- **GuardDuty** — основной детектор.
- **Security Hub** — единая очередь находок и их статусов.
- **EventBridge** — мгновенная маршрутизация на реакцию.
- **Lambda / Step Functions** — автоматические действия.
- **Systems Manager (SSM)** — выполнение действий на инстансах: снять память/диск для форензики, изолировать, применить патч, запустить runbook-документ.
- **AWS Backup** — быстрый откат к чистому состоянию при восстановлении.
- **CloudTrail / Detective / Athena** — расследование: что произошло и насколько широко.

#### Часть C. Роли и обязанности

Минимальный ролевой состав на инцидент:
- **Incident Commander** — единственный, кто принимает решения и координирует; не «копает» сам.
- **Технические ответственные (responders)** — выполняют локализацию/расследование.
- **Communications lead** — общается со стейкхолдерами, юристами, при необходимости — с регуляторами и клиентами.
- **Scribe** — ведёт хронологию действий (она же — будущие улики и постмортем).

Заранее назначенные дежурства (on-call) и матрица эскалации — кто подключается, если за N минут нет реакции.

#### Часть D. Коммуникационные процессы

- Выделенный канал инцидента (создаётся автоматически, например, Slack-канал по триггеру).
- Шаблоны сообщений (статус, влияние, ETA восстановления) — чтобы не сочинять под стрессом.
- Заранее согласованные с юристами критерии уведомления регуляторов: GDPR требует уведомления о значимой утечке персональных данных в течение 72 часов; в здравоохранении/финансах — свои сроки. Эти обязательства должны быть зашиты в playbook.
- **Важно для устойчивости:** канал оповещения не должен зависеть от скомпрометированной системы (out-of-band).

#### Часть E. Подготовка среды к инцидентам

**Стратегия развёртывания инструментов безопасности.** Все детективные сервисы (GuardDuty, Security Hub, Config, Inspector) включены заранее на всю организацию через делегированного администратора — во время инцидента их не «доустанавливают».

**Управление доступом для реагирования.** Заранее созданы **break-glass роли** для IR — с повышенными правами, обычно выключенные, включаемые только в инцидент, с обязательным MFA и полным логированием. Доступ выдаётся временно (через assume role с коротким сроком), а не постоянными ключами.

> **Новый термин — break-glass («разбить стекло»).** Аварийный доступ повышенного уровня, который в норме недоступен и активируется только в чрезвычайной ситуации, с громким аудитом каждого использования. Название — по аналогии с пожарной кнопкой за стеклом.

**Минимизация blast radius.**
- Изоляция через **отдельные аккаунты** под прод/данные.
- **Наименьшие привилегии** + **permission boundaries** (об этом в разделе 4).
- **Сегментация сети**: security groups, NACL, приватные подсети, отсутствие лишнего выхода в интернет.
- Отдельный **forensics-аккаунт**, куда складывают снимки скомпрометированных дисков для анализа в изоляции.

**Защитные механизмы (guardrails).** SCP, запрещающие отключать логирование/security-сервисы; обязательное шифрование; запрет публичных бакетов на уровне организации.

#### Часть F. Тестирование и валидация

План, который не тестировали, не работает. Способы:
- **Tabletop-учения** — команда «на бумаге» проходит сценарий: кто что делает. Дёшево, выявляет дыры в процессе.
- **Game Days** — управляемый реальный сценарий в нестрог-проде: запускают тестовую находку GuardDuty (есть функция генерации sample findings) и смотрят, сработала ли вся цепочка реакции.
- **Симуляция атак / purple team** — контролируемое воспроизведение техник атакующего.

> **Новый термин — GuardDuty sample findings.** Встроенная возможность сгенерировать «учебную» находку, чтобы проверить, что весь пайплайн (EventBridge → Lambda → оповещение) отрабатывает, не устраивая настоящую атаку.

**Критерии успеха** формулируют измеримо: находка зафиксирована за ≤ X минут; автолокализация отработала за ≤ Y минут; дежурный получил алерт; улики собраны; восстановление уложилось в целевые RTO/RPO.

> **Новые термины — RTO и RPO.** **RTO (Recovery Time Objective)** — за сколько мы обязаны восстановить сервис. **RPO (Recovery Point Objective)** — какой объём данных (по времени) допустимо потерять. Например, RPO=1ч означает «бэкап не старше часа», RTO=4ч — «поднять всё за 4 часа». Эти числа задают архитектуру бэкапов и репликации.

#### Часть G. Автоматизированное устранение и восстановление

Примеры автоматических действий по типу инцидента:
- **Скомпрометированный ключ IAM:** Lambda по находке GuardDuty прикрепляет к принципалу политику Deny / деактивирует ключ, снимает активные сессии.
- **Заражённый инстанс:** SSM/Lambda меняет security group на «карантинную» (без исходящего трафика), делает снимок диска и памяти в forensics-аккаунт, выводит инстанс из балансировщика.
- **Открытый бакет:** Lambda восстанавливает Block Public Access.
- **Восстановление:** поднять чистый ресурс из IaC + восстановить данные из AWS Backup; убедиться, что причина устранена, прежде чем возвращать в строй.

**Методы проверки восстановления:** прогон smoke-тестов, сверка целостности данных, подтверждение, что аномальная активность прекратилась, контроль повторного появления находок.

#### Проблемы внедрения и решения

- **Слишком агрессивная автоматизация** может «уронить» прод (например, изолировать здоровый инстанс по ложной находке). Решение: для разрушительных действий — режим «человек подтверждает» (human-in-the-loop через Step Functions с approval-шагом); полную автоматику — только для безопасных и точных триггеров.
- **Права на реагирование во время инцидента** — нельзя в панике раздавать админку. Решение: заранее заготовленные break-glass роли.
- **Расхождение прода и плана** — план устаревает. Решение: регулярные game days и хранение playbook как код.

#### Метрики успеха

- **MTTD** (Mean Time To Detect) — среднее время до обнаружения.
- **MTTR** (Mean Time To Respond/Recover) — до реакции/восстановления.
- Доля инцидентов, обработанных автоматически.
- Соблюдение RTO/RPO на учениях.

#### Непрерывное улучшение

Каждый инцидент и каждое учение завершается **постмортемом без поиска виноватых (blameless)**: что сработало, что нет, какие правки в playbook/архитектуру/автоматизацию. Эти правки возвращаются в IaC и в Config Rules.

#### Соответствие

IR-план — обязательное требование PCI DSS, HIPAA, SOC 2, ISO 27001. Аудиторам показывают: задокументированный план, доказательства тестирования (отчёты game days), сроки уведомления регуляторов, неизменяемые логи действий во время инцидента.

### 2.2. Полный цикл реагирования: обнаружение → восстановление

**Задача.** Глобальный финтех, мигрировавший критичные нагрузки в AWS. Среда: несколько аккаунтов через Organizations; приложения на EC2 и в контейнерах; чувствительные данные в RDS и DynamoDB; публичные приложения через API Gateway и CloudFront; микросервисы на Lambda и ECS. Нужна стратегия IR со стадиями обнаружение/анализ/локализация/устранение/восстановление, с учётом соответствия, непрерывности бизнеса, автоматизации, документации, координации и извлечения уроков. Привести конкретные сервисы, точки интеграции, автоматизацию рабочих процессов, процедуры реагирования и методы документирования.

Этот вопрос — про то же ядро, что 2.1, но требует пройти **по стадиям IR применительно к конкретным сервисам** среды. Сделаем именно так.

> **Поясним новые сервисы из условия.**
> - **Amazon RDS** — управляемые реляционные БД (PostgreSQL, MySQL и др.). AWS берёт на себя бэкапы, патчи, репликацию.
> - **Amazon DynamoDB** — управляемая NoSQL-база «ключ-значение/документ», масштабируется почти безгранично.
> - **Amazon API Gateway** — управляемый шлюз для публикации API: принимает HTTP-запросы, аутентифицирует, лимитирует, направляет на бэкенд (Lambda, контейнеры).
> - **Amazon CloudFront** — CDN (сеть доставки контента): кеширует и отдаёт контент с пограничных точек по всему миру близко к пользователю; заодно — точка применения WAF и защиты от DDoS.
> - **Amazon ECS / EKS** — оркестраторы контейнеров. **ECS** — собственный сервис AWS; **EKS** — управляемый Kubernetes. Контейнер — это легковесная упаковка приложения со всем окружением.

#### Стадия 1. Обнаружение (Detection)

Источники сигналов по каждому слою среды:
- **Аккаунты/IAM:** GuardDuty (аномальные API-вызовы, подозрительный assume role, утечки ключей).
- **EC2/контейнеры:** GuardDuty Runtime Monitoring (наблюдает за поведением процессов внутри EC2/EKS/ECS), Inspector (уязвимости в образах ECR и на инстансах).
- **RDS:** GuardDuty RDS Protection (аномальные логины к базе), аудит-логи БД.
- **DynamoDB / S3:** CloudTrail data events + GuardDuty S3 protection.
- **Веб-периметр (CloudFront/API Gateway):** WAF-логи и метрики, AWS Shield (DDoS), CloudWatch-алармы на всплески 4xx/5xx.

> **Новый термин — GuardDuty Runtime Monitoring.** Расширение GuardDuty, которое через лёгкий агент видит, что *внутри* контейнеров и инстансов происходит на уровне ОС (подозрительные процессы, обращения к файлам, сетевые соединения) — а не только по логам снаружи.

> **Новый термин — AWS Shield.** Защита от DDoS-атак. **Shield Standard** включён бесплатно для всех. **Shield Advanced** — платный, с расширенной защитой L3/L4/L7, поддержкой команды реагирования AWS (SRT) и компенсацией расходов от атаки.

Всё это сводится в **Security Hub** (единая очередь) и маршрутизируется **EventBridge**.

#### Стадия 2. Анализ (Analysis / Triage)

- **Amazon Detective** строит граф: какой принципал, с каких IP, какие ресурсы и API трогал — быстрый ответ на «насколько всё плохо».
- **CloudTrail + Athena** — точечные SQL-запросы по истории действий.
- **VPC Flow Logs** — куда ходил скомпрометированный ресурс по сети.
Цель стадии — определить тип, масштаб (scope) и затронутые данные (важно для обязательств по уведомлению).

#### Стадия 3. Локализация (Containment)

Цель — остановить распространение, *сохранив улики*. По типам:
- **EC2/контейнер:** перевести в карантинную security group (обрезать исходящий трафик), вывести из ALB/целевой группы, для ECS/EKS — остановить/закордонировать задачу/под. Снять снимки диска (EBS snapshot) и памяти в forensics-аккаунт.
- **IAM-компрометация:** деактивировать ключ, приложить explicit Deny, отозвать активные сессии (`aws iam` + ревокация сессий ролей через политику с условием по времени выдачи).
- **RDS/DynamoDB:** ограничить доступ на сетевом уровне и через политики, снять снапшот для расследования, при необходимости — point-in-time снимок.
- **Веб-периметр:** добавить блокирующие правила в WAF (IP/паттерн), включить/усилить Shield Advanced.

Локализацию ценных «быстрых» действий автоматизируют через EventBridge → Lambda; разрушительное — через Step Functions с approval-шагом.

#### Стадия 4. Устранение (Eradication)

Убрать первопричину: пропатчить уязвимость (Inspector укажет какую), убрать вредонос/бэкдор, заменить скомпрометированные секреты (ротация в Secrets Manager — раздел 5.3), закрыть неверную конфигурацию (через Config + remediation). Скомпрометированные вычисления обычно **не лечат, а пересоздают** из чистого образа/IaC (immutable-подход надёжнее «чистки»).

#### Стадия 5. Восстановление (Recovery)

- Поднять чистую инфраструктуру из **IaC**.
- Восстановить данные из **AWS Backup** / снапшотов / point-in-time recovery (RDS, DynamoDB).
- Для непрерывности бизнеса — заранее настроенная **мульти-AZ** (отказ дата-центра) и при необходимости **мульти-региональная** репликация и план DR с целевыми RTO/RPO.
- Перед возвратом в прод — проверить, что причина устранена и аномалии исчезли (иначе повторная компрометация).

#### Стадия 6. Извлечение уроков (Lessons Learned)

Blameless-постмортем, обновление playbook, IaC, Config Rules и автоматизации. Метрики (MTTD/MTTR) сравнивают с прошлыми инцидентами.

#### Сквозные требования из условия

- **Соответствие:** неизменяемые логи (Object Lock), сроки уведомления (GDPR 72ч и отраслевые), документированные процедуры, доказательства тестирования.
- **Непрерывность бизнеса:** мульти-AZ/мульти-регион, AWS Backup, протестированный DR.
- **Автоматизация:** EventBridge + Lambda/Step Functions + SSM (детали выше).
- **Документация:** автоматический сбор хронологии (CloudTrail + scribe), тикет инцидента, постмортем.
- **Координация команды:** Incident Commander + роли + on-call + выделенный канал.

#### Точки интеграции между сервисами (карта)

```
GuardDuty/Inspector/Config/WAF ──► Security Hub ──► EventBridge
                                                        │
                  ┌─────────────────────────────────────┼──────────────────────────┐
                  ▼                                      ▼                          ▼
                 SNS                              Lambda (быстрая              Step Functions
            (PagerDuty/Slack)                     локализация)               (playbook + approval)
                                                       │                          │
                                                       ▼                          ▼
                                                  SSM (действия            Detective/Athena
                                                  на инстансах)            (анализ), AWS Backup
                                                                            (восстановление)
```

## Раздел 3. Безопасность на периферии сети и защита вычислений

> **Что значит «безопасность на периферии» (edge security).** «Периферия» (edge) — это граница, где ваши сервисы встречаются с внешним миром: пользователи, интернет, сторонние интеграции, IoT-устройства. Здесь происходит большинство атак (DDoS, веб-эксплойты, бот-трафик), поэтому защита периметра — отдельная дисциплина: CDN, WAF, защита от DDoS, контроль API, защита публичных данных.

### 3.1. Edge-безопасность для финансовой организации

**Задача.** Глобальный финтех с миллионами транзакций в день расширяется на международный уровень. Среда: несколько дистрибутивов CloudFront, API через API Gateway, S3 с чувствительными данными клиентов, IoT-устройства, передающие финансовые данные, требования интеграции со сторонними security-сервисами и соответствия множеству международных норм. Спроектировать комплексную edge-стратегию с учётом соответствия, влияния на производительность, масштабируемости, финансов и операционных накладных. Привести конфигурации, этапы, мониторинг, troubleshooting, документацию.

#### Слой 1. CloudFront как защищённая точка входа

Весь публичный трафик заводят через **CloudFront** (CDN), а не напрямую на бэкенды. Это даёт сразу несколько выгод безопасности:
- Бэкенды (ALB, S3, API Gateway) скрыты от прямого доступа — клиенты видят только CloudFront.
- На CloudFront навешивают WAF, Shield и TLS — единая точка контроля.
- Кеширование и близость к пользователю снижают задержку (важно для финтеха) и поглощают часть нагрузки.

**Защита origin (бэкенда) от обхода CDN.** Чтобы никто не ходил на бэкенд мимо CloudFront:
- Для S3 — **Origin Access Control (OAC)**: бакет закрыт от всех, кроме конкретного CloudFront-дистрибутива.
- Для кастомных origin — секретный заголовок, который проверяет WAF/ALB, плюс ограничение по диапазонам IP CloudFront.

> **Новый термин — Origin Access Control (OAC).** Механизм, при котором приватный S3-бакет отдаёт контент *только* через ваш CloudFront, а прямые запросы к бакету запрещены. Так чувствительные данные не «висят» в открытом доступе.

#### Слой 2. AWS WAF — фильтрация приложенческих атак

На CloudFront и API Gateway вешают **WAF** со связкой правил:
- **AWS Managed Rules** — готовые наборы от AWS: Core rule set (OWASP-подобные атаки), известные плохие IP, защита от SQLi/XSS, anonymous IP list (Tor/прокси).
- **Rate-based rules** — лимит запросов с одного источника (анти-брутфорс, анти-скрейпинг).
- **Bot Control** — отсев ботов.
- **Geo-match** — ограничение/приоритезация по странам (для соответствия и снижения шума).
- **Кастомные правила** под бизнес (например, защита транзакционных эндпоинтов).

> **Новый термин — OWASP Top 10.** Список самых распространённых уязвимостей веб-приложений (инъекции, сломанная аутентификация, XSS и т.д.). Managed Rules WAF во многом закрывают именно их.

#### Слой 3. Защита от DDoS

- **AWS Shield Standard** — бесплатно, базовая защита L3/L4 на CloudFront/Route 53.
- **AWS Shield Advanced** — для финтеха с критичной доступностью оправдан: расширенная защита, доступ к **Shield Response Team (SRT)**, защита от роста счёта при атаке (cost protection), расширенные метрики. Связывается с WAF для автоматического реагирования на L7-атаки.

#### Слой 4. Безопасность API (API Gateway)

- **Аутентификация/авторизация:** Cognito user pools или Lambda authorizers / IAM-авторизация (детали — раздел 4).
- **Throttling и usage plans** — лимиты запросов на клиента (защита от злоупотреблений и каскадных отказов).
- **Mutual TLS (mTLS)** — взаимная проверка сертификатов клиента и сервера; критично для партнёрских и межбанковских API.
- **Приватные API** (private API Gateway через VPC endpoint) для внутренних интеграций — не выходят в интернет вовсе.
- WAF перед API для тех же приложенческих фильтров.

> **Новый термин — mutual TLS (mTLS).** Обычный TLS проверяет только сервер. При mTLS *обе* стороны предъявляют сертификаты и проверяют друг друга. Это сильная аутентификация машина-машина, стандарт для финансовых партнёрских интеграций.

#### Слой 5. Защита данных в S3

- **Block Public Access** на уровне аккаунта и бакета (запрет случайной публикации).
- Доступ только через **OAC** (см. выше) или приватные пути.
- **Шифрование в покое** (SSE-KMS) и **в движении** (TLS-only bucket policy с условием `aws:SecureTransport`).
- **Amazon Macie** — сервис, который машинным обучением находит в S3 чувствительные данные (номера карт, PII) и сообщает, если они лежат не там или открыты.

> **Новый термин — PII (Personally Identifiable Information).** Персональные данные, по которым можно идентифицировать человека (имя, номер карты, паспорт). Их защита — предмет GDPR, PCI DSS и др. **Macie** автоматически их обнаруживает в S3.

#### Слой 6. Безопасность IoT

Для устройств, передающих финансовые данные, используют **AWS IoT Core**:
- Каждому устройству — уникальный сертификат X.509 (никаких общих паролей).
- **IoT policies** — наименьшие привилегии: устройство публикует только в свой топик.
- **AWS IoT Device Defender** — аудит и обнаружение аномалий поведения устройств (вдруг устройство шлёт нетипичный объём/в нетипичное место).
- TLS на всех соединениях; ротация/отзыв сертификатов при компрометации.

> **Новый термин — AWS IoT Core / Device Defender.** IoT Core — управляемый брокер сообщений для устройств (по MQTT и др.) с аутентификацией по сертификатам. Device Defender — слой безопасности поверх: проверяет конфигурации устройств и ловит аномалии.

#### Слой 7. Интеграция со сторонними security-сервисами

- Логи WAF/CloudFront → **Kinesis Firehose** → внешний SIEM.
- Находки → через **Security Hub** (ASFF) в сторонние системы.
- Сторонние WAF/threat-feed подключают как managed rules или через автоматизацию обновления списков IP.

#### Соответствие, производительность, масштаб, финансы, операционка

- **Соответствие:** PCI DSS прямо требует WAF перед платёжными приложениями, шифрование, ограничение доступа; GDPR — защиту PII и контроль трансграничной передачи (geo-controls помогают). Всё это закрывается слоями выше.
- **Производительность:** CloudFront снижает задержку кешированием и близостью к пользователю; WAF добавляет минимальные миллисекунды; TLS-терминация на edge разгружает бэкенд. Баланс: не злоупотреблять тяжёлыми правилами WAF на латентно-чувствительных путях.
- **Масштабируемость:** CloudFront/WAF/Shield — глобальные управляемые сервисы, масштабируются автоматически под миллионы запросов.
- **Финансы:** основные статьи — Shield Advanced (фиксированная немалая плата), WAF (за правила и запросы), CloudFront (за трафик), Macie (за объём сканирования). Оптимизируют кешированием (меньше трафика к origin), разумным набором правил, выборочным Macie.
- **Операционные накладные:** управляемые сервисы снижают рутину; всё конфигурируется через IaC; правила WAF — в версионируемых наборах.

#### Этапы реализации (порядок)

1. Завести трафик через CloudFront, закрыть прямой доступ к origin (OAC/секретный заголовок).
2. Подключить TLS (ACM-сертификаты — раздел 5) и enforce HTTPS.
3. Навесить WAF (managed + rate-based), сначала в режиме count, потом block.
4. Включить Shield (Advanced для критичных).
5. Закрыть S3 (Block Public Access, шифрование, Macie).
6. Усилить API Gateway (authz, throttling, mTLS для партнёров).
7. Защитить IoT (сертификаты, Device Defender).
8. Подключить логи и интеграции с SIEM, настроить мониторинг.

#### Мониторинг и troubleshooting

- **Мониторинг:** метрики WAF (заблокированные/разрешённые), Shield (DDoS-метрики), CloudFront (коды ответов, cache hit ratio), алармы на всплески 4xx/5xx.
- **Troubleshooting:** WAF в режиме **count** перед **block** — чтобы увидеть, что правило заблокирует, не ломая легитимный трафик (борьба с ложными срабатываниями); анализ WAF-логов в Athena; проверка, что origin доступен только через CDN.

> **Совет.** Любое новое правило WAF сначала ставят в режим подсчёта (count), смотрят, кого бы оно зацепило, и только убедившись — переводят в блокировку. Это главный приём против «положили легитимных пользователей».

#### Документация

Архитектурная диаграмма edge-слоёв, реестр WAF-правил с обоснованием каждого, runbook на DDoS и на «ложно заблокировали клиента», маппинг контролей на требования PCI/GDPR.

### 3.2. Безопасность вычислительных нагрузок (EC2, контейнеры, serverless, GenAI)

**Задача.** Глобальная компания модернизирует инфраструктуру: прод-нагрузки на EC2; контейнеры на ECS и EKS; serverless на Lambda; DevOps-пайплайны непрерывного деплоя; недавно внедрённые приложения генеративного ИИ; строгие PCI DSS и HIPAA. Нужна комплексная стратегия безопасности вычислений с учётом соответствия, влияния на операции, автоматизации/масштабируемости, ограничений ресурсов, непрерывности бизнеса. Привести настройки инструментов, процедуры внедрения, мониторинг, процессы remediation, документацию.

Защита вычислений = защита того, *на чём* и *как* выполняется код. Разберём по типам нагрузок плюс общий конвейер.

#### Общий принцип для всех типов: identity вместо паролей

Ни одна нагрузка не должна носить «вшитые» ключи. Вместо этого:
- **EC2** получает права через **instance profile** (роль, привязанную к инстансу).
- **ECS/EKS** — через **task role** / **IRSA** (IAM Roles for Service Accounts) — каждому контейнеру/поду своя роль.
- **Lambda** — через **execution role**.
Это и есть наименьшие привилегии на уровне рабочей нагрузки.

> **Новый термин — IRSA (IAM Roles for Service Accounts).** В EKS позволяет привязать IAM-роль к конкретному Kubernetes service account, то есть выдать права отдельному поду, а не всей ноде. Минимизирует blast radius внутри кластера.

#### EC2

- **Inspector** непрерывно сканирует ОС/пакеты на уязвимости (нужно для PCI/HIPAA).
- **Systems Manager Patch Manager** — автоматическое управление патчами по расписанию.
- **SSM Session Manager** вместо SSH: доступ к инстансу через браузер/CLI без открытого 22-го порта и без bastion, с полным логированием сессий. Резко уменьшает поверхность атаки.
- **GuardDuty Runtime Monitoring** — поведенческий контроль на уровне ОС.
- Шифрование томов **EBS** через KMS; **IMDSv2** обязателен (защита от подделки запросов к метаданным).
- Immutable-подход: «золотой» образ (AMI) собирается в пайплайне, прошедшем сканирование; инстансы не патчат вручную, а пересоздают.

> **Новый термин — IMDSv2.** Instance Metadata Service v2 — защищённая версия сервиса метаданных EC2 (откуда инстанс берёт свои временные креды). v2 требует токен и закрывает класс атак SSRF, через который раньше воровали роли инстансов. Должен быть принудительно включён.

> **Новый термин — bastion host.** Защищённый «прыжковый» сервер для доступа во внутреннюю сеть. SSM Session Manager позволяет от него отказаться вовсе.

#### Контейнеры (ECS / EKS)

- **Сканирование образов в ECR** (Inspector / ECR scanning) на этапе пуша — не допускать уязвимые образы в прод.
- **Подписывание образов** и допуск только доверенных (image provenance).
- **Минимальные базовые образы**, не root внутри контейнера, read-only файловая система.
- **EKS:** сетевые политики (NetworkPolicy), Pod Security Standards, RBAC по наименьшим привилегиям, приватные кластеры, IRSA.
- **ECS:** task role на задачу, awsvpc-режим сети, секреты из Secrets Manager (не в переменных окружения открыто).
- **GuardDuty EKS/Runtime Protection** — обнаружение угроз в кластере.
- Сегментация: прод-контейнеры в приватных подсетях, строгие security groups.

> **Новый термин — RBAC (Role-Based Access Control).** Управление доступом на основе ролей — кто какие действия может выполнять. В Kubernetes RBAC определяет, что может делать пользователь/сервис в кластере.

#### Serverless (Lambda)

- Минимальная execution role на функцию (часто — отдельная роль на функцию).
- Секреты — из Secrets Manager / Parameter Store, не в коде.
- Сканирование зависимостей (Inspector for Lambda) — уязвимости в библиотеках.
- Функции в VPC, если нужен доступ к приватным ресурсам; жёсткий контроль исходящих соединений.
- Лимиты конкуррентности (защита от runaway-затрат и от использования в атаке).
- Подпись кода (Lambda code signing) — допуск только доверенных артефактов.

#### DevOps-пайплайны (защита самого конвейера деплоя)

Конвейер — лакомая цель: кто контролирует деплой, контролирует прод. Поэтому встраивают безопасность *в* пайплайн (**shift-left**, **DevSecOps**):
- **SAST** (статический анализ кода), **SCA** (анализ зависимостей), **secret scanning** (поиск утёкших ключей в коде), **IaC-сканирование** (checkov/cfn-guard) — на каждый коммит/PR.
- **Сканирование образов** перед публикацией.
- Пайплайн **не имеет долгоживущих ключей** — использует роли с OIDC (например, GitHub Actions ↔ IAM через OIDC, без хранения секретов).
- Разделение прав: тот, кто пишет код, не обязательно тот, кто аппрувит прод-деплой.
- Артефакты подписываются, провенанс проверяется.

> **Новые термины — shift-left / DevSecOps.** «Сдвиг влево» — переносить проверки безопасности как можно раньше в цикл разработки (на этап кода и сборки), а не находить дыры уже в проде. DevSecOps — встраивание безопасности в каждый этап DevOps-конвейера автоматически.

> **Новые термины — SAST / SCA.** **SAST** (Static Application Security Testing) — анализ исходного кода на уязвимые паттерны. **SCA** (Software Composition Analysis) — проверка сторонних библиотек на известные уязвимости.

> **Новый термин — OIDC для CI/CD.** Способ, которым внешний CI (GitHub Actions, GitLab) получает временную IAM-роль через доверие к OIDC-провайдеру — без хранения постоянных AWS-ключей в настройках пайплайна. Это устраняет самый частый источник утечек.

#### Приложения генеративного ИИ (GenAI)

Относительно новый класс нагрузок со своими рисками. Если используется **Amazon Bedrock** (управляемый доступ к фундаментальным моделям):
- **Bedrock Guardrails** — фильтры на ввод/вывод: блокировка вредного контента, **защита от prompt injection**, маскирование PII в промптах и ответах.
- Изоляция данных: на запросах к моделям не утекают чувствительные данные за пределы периметра; приватные эндпоинты (PrivateLink).
- Логирование вызовов модели (model invocation logging) для аудита.
- Наименьшие привилегии к данным, к которым модель обращается (особенно в RAG-сценариях — модель не должна видеть больше, чем положено пользователю).
- Контроль затрат и лимиты (защита от «выкачивания»/злоупотребления).

> **Новые термины — prompt injection и Bedrock Guardrails.** **Prompt injection** — атака, когда злоумышленник через входной текст заставляет LLM игнорировать инструкции или выдать лишнее. **Bedrock Guardrails** — управляемый слой AWS, фильтрующий вход/выход моделей: запрещённые темы, утечка PII, попытки инъекции.

> **Новый термин — RAG (Retrieval-Augmented Generation).** Подход, когда модель отвечает, подтягивая данные из вашей базы знаний. Риск безопасности: модель не должна отдавать пользователю данные, к которым у него нет доступа — нужен контроль доступа на уровне извлечения.

#### Соответствие (PCI DSS / HIPAA)

- Шифрование everywhere (KMS at rest, TLS in transit).
- Сегментация среды с данными карт/PHI (отдельные подсети/аккаунты — уменьшение PCI-scope).
- Управление уязвимостями (Inspector + патчинг) — прямое требование.
- Полный аудит (CloudTrail, логи доступа), неизменяемость.
- Контроль доступа по наименьшим привилегиям, MFA.
- Для HIPAA — заключённый с AWS BAA и использование только HIPAA-eligible сервисов.

> **Новый термин — BAA (Business Associate Agreement).** Соглашение, обязательное по HIPAA между организацией и поставщиком (AWS), регулирующее обращение с PHI. Без него использовать AWS для PHI нельзя.

#### Мониторинг, remediation, непрерывность, документация

- **Мониторинг:** GuardDuty (все типы), Inspector (уязвимости), Config (конфигурации), Security Hub (агрегация), Runtime Monitoring (поведение). Дашборды и алармы — как в разделе 1.
- **Remediation:** автоматическая через EventBridge → Lambda/SSM (изоляция, патч, откат конфигурации); для разрушительного — с подтверждением.
- **Непрерывность бизнеса:** мульти-AZ для всех уровней, автоскейлинг, AWS Backup, отрепетированный DR.
- **Ограничения ресурсов:** приоритезация — сначала закрыть критичные пути (данные карт/PHI, публичный периметр), потом остальное; максимально опираться на управляемые сервисы, чтобы небольшая команда справлялась.
- **Документация:** baseline-конфигурации как код, реестр контролей с маппингом на PCI/HIPAA, runbook-и на типовые находки.

> **Примечание.** Третий вопрос Раздела 3 в исходнике дословно повторяет вопрос 1.3 (диагностика проблем логирования и мониторинга) — его полный разбор смотрите выше в [пункте 1.3](#13-диагностика-проблем-логирования-и-мониторинга).

## Раздел 4. Аутентификация и авторизация (Identity)

> **Аутентификация vs авторизация — не путать.** **Аутентификация (authN)** — «кто ты» (проверка личности). **Авторизация (authZ)** — «что тебе можно» (проверка прав). Сначала система убеждается, кто вы, потом решает, что вам разрешено. В AWS authN для людей всё чаще идёт через единый вход (SSO), а authZ выражается в IAM-политиках.

### 4.1. Стратегия аутентификации для транснациональной корпорации

**Задача.** Крупная корпорация переносит всю инфраструктуру в AWS. Нужны: безопасный доступ для сотрудников в офисах по всему миру, интеграция с существующими провайдерами идентичности, временный доступ для подрядчиков, аутентификация для внутренних и внешних приложений, строгий аудит и MFA. Спроектировать, внедрить и поддерживать комплексную стратегию аутентификации: для людей, приложений и систем; временные креды; обработка проблем аутентификации. Обосновать выбор сервисов и обеспечить безопасность/масштабируемость.

Ключевая идея: **разные типы субъектов — разные механизмы.** Не существует «одного логина на всё». Разделим на три мира: люди-сотрудники, внешние пользователи (клиенты приложений), и машины (приложения/системы).

#### Мир 1. Сотрудники (workforce identity)

**AWS IAM Identity Center** (бывший AWS SSO) — центральный сервис единого входа для людей в AWS.
- Подключается к существующему **провайдеру идентичности (IdP)** компании — Active Directory, Azure AD/Entra ID, Okta, Ping — через **SAML/OIDC** и **SCIM** (для автоматической синхронизации пользователей и групп).
- Сотрудник логинится один раз своими корпоративными учётными данными и получает доступ к нужным аккаунтам AWS и приложениям с уже назначенными ролями.
- **Permission sets** — наборы прав, которые Identity Center раздаёт по группам в нужных аккаунтах. Управление централизованное.
- Все сессии — **временные** (через assume role под капотом), постоянных IAM-пользователей для людей не создают вообще.

> **Новые термины — IdP, SAML, OIDC, SCIM.**
> - **IdP (Identity Provider)** — система, которая хранит учётки и подтверждает личность (AD, Okta, Entra ID).
> - **SAML / OIDC** — протоколы федерации: они позволяют AWS доверять чужому IdP, чтобы пользователь логинился там, а доступ получал здесь. SAML — старший корпоративный стандарт, OIDC — современный поверх OAuth 2.0.
> - **SCIM** — протокол автоматической синхронизации пользователей/групп из IdP в AWS (завели сотрудника в AD — он появился в AWS; уволили — исчез доступ).

> **Почему так, а не IAM-пользователи.** IAM-пользователи с паролями и долгоживущими ключами для людей — антипаттерн: их трудно отзывать, легко потерять, тяжело аудировать в масштабе. Федерация + временные сессии решают это: «источник правды» о людях — корпоративный IdP, а AWS лишь доверяет ему.

**MFA.** Многофакторная аутентификация обязательна. Идеально — устойчивые к фишингу факторы (FIDO2/passkeys, аппаратные ключи). Enforce на уровне IdP и/или Identity Center.

> **Новый термин — MFA (Multi-Factor Authentication).** Подтверждение личности не одним фактором (пароль), а несколькими: что-то, что вы знаете (пароль) + что-то, что у вас есть (телефон/ключ) + что-то, чем вы являетесь (биометрия). Даже укравший пароль не пройдёт без второго фактора. **FIDO2/passkey** — современный фактор, устойчивый к фишингу.

#### Мир 2. Внешние пользователи приложений (customer identity)

Для клиентов публичных приложений — **Amazon Cognito**:
- **User pools** — каталог пользователей приложения с регистрацией, входом, MFA, восстановлением пароля, верификацией.
- Поддержка **социального входа** (Google, Apple) и федерации с корпоративными IdP.
- Выдаёт токены (OIDC/JWT), которыми приложение авторизует запросы к API Gateway/бэкенду.
- **Identity pools** — обменивают токен на временные AWS-креды, если клиентскому приложению нужен прямой (ограниченный) доступ к AWS-ресурсу.

> **Новый термин — Amazon Cognito.** Сервис управления идентичностью *конечных пользователей приложений* (не сотрудников). Берёт на себя регистрацию/вход/MFA и выдачу токенов, чтобы вы не писали свою систему аутентификации.

> **Новый термин — JWT (JSON Web Token).** Подписанный токен с информацией о пользователе и его правах. Приложение проверяет подпись и доверяет содержимому, не дёргая базу на каждый запрос.

#### Мир 3. Машины: приложения и системы

- **Внутри AWS** — только **роли** (instance profile, task role/IRSA, Lambda execution role). Никаких ключей в коде.
- **Вне AWS** (внешний CI, локальный сервис, партнёр) — **IAM Roles Anywhere** (доступ по сертификатам X.509 без долгоживущих ключей) или OIDC-федерация. Партнёрский доступ — через **cross-account роли** с `ExternalId`.
- Секреты приложений (пароли БД, API-ключи) — в **Secrets Manager** с автоматической ротацией (раздел 5.3).

> **Новый термин — IAM Roles Anywhere.** Позволяет нагрузкам *вне* AWS получать временные AWS-креды, предъявляя сертификат X.509 от доверенного УЦ. Убирает необходимость хранить статические access keys на он-преме.

> **Новый термин — cross-account роль с ExternalId.** Способ дать стороннему аккаунту/партнёру доступ к вашим ресурсам: вы создаёте роль, которой доверяет их аккаунт, а `ExternalId` (общий секрет) защищает от атаки «confused deputy». Партнёр получает временную сессию, а не ваши ключи.

#### Временный доступ для подрядчиков

- Подрядчиков заводят в IdP/Identity Center с **ограниченным сроком** и узким **permission set**.
- Для разовых задач — **assume role на минуты/часы**, не постоянный доступ.
- Автоматическое снятие доступа по окончании контракта через SCIM/деактивацию в IdP.
- Всё под MFA и полным аудитом.

#### Обработка проблем аутентификации

- **CloudTrail** фиксирует каждую попытку входа/assume role — источник для расследования «почему не пускает» и «кто это логинился».
- GuardDuty ловит аномальные входы (нетипичная страна, перебор).
- Типовые проблемы: рассинхрон SCIM (пользователь есть в AD, нет в AWS), просроченные сессии, неверный trust policy у роли, конфликт SCP/permission boundary. Диагностируют по `AccessDenied` в CloudTrail и симулятору политик.

#### Безопасность и масштабируемость

- Источник правды о людях — корпоративный IdP; AWS-доступ — производный и временный → легко масштабировать на тысячи сотрудников и десятки аккаунтов.
- Identity Center раздаёт permission sets по всем аккаунтам организации централизованно.
- Cognito и Identity Center — управляемые и эластичные.
- Аудит и MFA — сквозные.

### 4.2. Комплексная стратегия IAM с временным доступом и аудитом

**Задача.** Быстрорастущая глобальная компания в большой цифровой трансформации на AWS. Требования: множеству бизнес-юнитов в разных странах нужен безопасный доступ к AWS; командам разработки — деплоить, сохраняя безопасность; подрядчикам — временный доступ к конкретным ресурсам; клиентским приложениям — надёжная аутентификация; весь доступ — аудируемый и соответствующий международным нормам; масштабируемость; интеграция legacy-систем на переходный период; быстрое выявление и устранение инцидентов. Описать: архитектуру IAM, реализацию least privilege, управление временным доступом и кредами, мониторинг/troubleshooting, выявление и устранение проблем безопасности, безопасность при росте. Назвать сервисы и обосновать; описать автоматизацию обслуживания и реагирования.

Этот вопрос — «старший брат» 4.1: те же строительные блоки, но акцент на **архитектуре, least privilege, автоматизации и аудите в масштабе.** Базовые механизмы (Identity Center, Cognito, роли, MFA, федерация) описаны в 4.1 — здесь не повторяем, а достраиваем.

#### Архитектура управления идентичностью и доступом

Многоуровневая модель:
1. **Организация (AWS Organizations)** задаёт границы через **SCP** — что вообще можно в аккаунтах (раздел 6).
2. **IAM Identity Center** — единая точка входа сотрудников во все аккаунты, **permission sets** по бизнес-юнитам и ролям.
3. **Cognito** — отдельный контур для клиентов приложений.
4. **Роли (а не пользователи)** — для всех машинных доступов.
5. **Federation** с корпоративным IdP для людей; **OIDC/Roles Anywhere** для внешних машин.

«Источник правды» по людям — IdP; по машинам — роли; по клиентам — Cognito. Постоянных IAM-пользователей и долгоживущих ключей избегают как класс.

#### Реализация наименьших привилегий (least privilege) на практике

Это не «дать поменьше один раз», а процесс:
- **Старт с минимума, расширение по необходимости.** Начинают с узких политик; недостающие права добавляют по фактической потребности, видимой в CloudTrail.
- **IAM Access Analyzer (policy generation)** — генерирует политику на основе *реально использованных* за период действий из CloudTrail. Готовый способ ужать раздутые права до фактических.
- **Permission boundaries** — «потолок» прав для роли/пользователя: даже если кто-то приложит более широкую политику, реальные права не выйдут за boundary. Важно, чтобы команды, которым позволено создавать роли (для деплоя), не могли выдать больше, чем им самим можно.
- **SCP** — потолок на уровне всего аккаунта/OU.
- **Access Analyzer (unused access)** — регулярно находит выданные, но не используемые права и роли → их урезают/удаляют.
- **Атрибутный доступ (ABAC)** — права по тегам (например, «доступ только к ресурсам своего проекта по тегу `Project`»). Масштабируется лучше, чем плодить роли под каждый случай.

> **Новые термины — permission boundary, ABAC.**
> - **Permission boundary** — максимально допустимый набор прав для конкретной сущности. Эффективные права = пересечение её политик и boundary. Защищает от эскалации привилегий при делегировании.
> - **ABAC (Attribute-Based Access Control)** — доступ на основе атрибутов/тегов, а не на основе перечисления ролей. «Можно трогать ресурсы с тегом отдела = твой отдел» — одно правило вместо сотни ролей.

#### Управление временным доступом и кредами

- Все человеческие сессии — короткоживущие через Identity Center/STS.
- Подрядчики — ограниченный по времени permission set + assume role на задачу.
- Машины — роли; внешние машины — Roles Anywhere/OIDC.
- Секреты — **Secrets Manager** с ротацией; параметры — **SSM Parameter Store**.
- **Никаких долгоживущих access keys**; если где-то в legacy неизбежны — короткая ротация, хранение в Secrets Manager, мониторинг использования через CloudTrail и Access Analyzer.

> **Новый термин — AWS STS (Security Token Service).** Сервис, выдающий *временные* учётные данные (на минуты–часы) при assume role или федерации. Фундамент всего «временного доступа» в AWS.

#### Интеграция legacy-систем на переходный период

- Старые системы, умеющие только SAML, — федерируют через Identity Center/IAM SAML.
- Системы со статическими ключами — оборачивают в Secrets Manager + ротация, параллельно переводя на роли/Roles Anywhere.
- Постепенная миграция: сначала федерация поверх legacy, затем вывод legacy из эксплуатации.

#### Мониторинг и troubleshooting

- **CloudTrail** — каждый вход, assume role, изменение политики.
- **Access Analyzer** — внешний доступ и неиспользуемые права.
- **GuardDuty** — аномальная identity-активность (нетипичные регионы, перебор, подозрительный assume role).
- **IAM credential report / last accessed** — какие креды/права давно не используются.
- Troubleshooting `AccessDenied`: смотреть в CloudTrail конкретный отказ → проверять цепочку (identity policy → resource policy → permission boundary → SCP → session policy) в симуляторе.

#### Выявление и устранение проблем безопасности (автоматизация)

Привычная связка под identity-инциденты:
```
GuardDuty/Access Analyzer/Config ──► EventBridge ──► Lambda/Step Functions
   (утёкший ключ,                                        │
    публичная роль,                                      ├─ деактивировать ключ
    эскалация прав)                                      ├─ приложить Deny
                                                         ├─ отозвать сессии
                                                         └─ уведомить SOC (SNS)
```
- **Config Rules** ловят дрейф: появился IAM-пользователь там, где должны быть только роли; роль стала слишком широкой; MFA выключен — авто-remediation или алерт.
- Автоматическое снятие неиспользуемых прав по данным Access Analyzer.

#### Безопасность при росте

- Всё централизовано в Organizations + Identity Center → новые аккаунты/команды подключаются по шаблону.
- ABAC и permission sets масштабируются без взрывного роста числа ролей.
- Guardrails (SCP) и Config Rules применяются автоматически ко всем новым аккаунтам.
- IaC для ролей/политик → ревью и аудит изменений доступа.

#### Соответствие

Аудируемость (CloudTrail, неизменяемые логи), MFA, least privilege, временный доступ, разделение обязанностей (separation of duties) — прямые требования PCI DSS, SOC 2, ISO 27001, GDPR. Архитектура выше закрывает их по построению.

## Раздел 5. Шифрование и защита данных

> **Два состояния данных, которые шифруют.** **At rest (в покое)** — когда данные лежат на диске/в хранилище (S3, EBS, RDS). **In transit (в движении)** — когда данные передаются по сети между компонентами. Хорошая защита покрывает оба, плюс отдельно — управление ключами (кто и как может расшифровать).

> **Базовый сервис — AWS KMS (Key Management Service).** Хранит и контролирует криптографические ключи. Сами секретные ключи не покидают KMS (хранятся в защищённых аппаратных модулях). Сервисы AWS обращаются к KMS, чтобы зашифровать/расшифровать данные, а доступ к ключам управляется политиками. **Envelope encryption (конвертное шифрование):** данные шифруются быстрым data key, а сам data key шифруется мастер-ключом из KMS — так совмещают скорость и контроль.

> **Альтернатива/дополнение — AWS CloudHSM.** Выделенный аппаратный модуль безопасности (HSM) под вашим единоличным контролем. Нужен, когда регуляторика требует, чтобы ключами управляли только вы (single-tenant), либо нужен FIPS 140-2 Level 3 и собственный контроль. Дороже и сложнее KMS; используется точечно.

### 5.1. Шифрование «в движении» и безопасный доступ для финтеха

**Задача.** Глобальный финтех на AWS с чувствительными финансовыми данными. Требования: всё шифруется в движении между всеми сервисами; торговым приложениям нужна сверхнизкая задержка при сохранении безопасности; ML-модели обрабатывают чувствительные данные клиентов; аналитика — большие объёмы транзакций; удалённым сотрудникам — безопасный доступ к внутренним ресурсам; партнёрам — к определённым приложениям; соответствие финансовым нормам; мультирегиональная масштабируемость; единообразие контролей; минимум влияния на производительность. Описать архитектуру шифрования (клиентские приложения, межсервисная связь, контейнеры, big data, ML), подход к TLS, жизненному циклу сертификатов, частной сети, удалённому доступу, партнёрской связи; контроль соответствия шифрования, ротацию ключей, влияние на производительность, высокую доступность.

Фокус этого вопроса — **encryption in transit и безопасный сетевой доступ.** Разберём по требуемым доменам.

#### Архитектура шифрования по доменам

**Клиентские (публичные) приложения.**
- TLS 1.2/1.3 на CloudFront/ALB/API Gateway; сертификаты из **AWS Certificate Manager (ACM)**.
- HSTS, современные cipher suites, редирект HTTP→HTTPS.
- TLS-терминация на edge (CloudFront) для низкой задержки + повторное шифрование до origin при необходимости.

> **Новый термин — AWS Certificate Manager (ACM).** Сервис, который бесплатно выпускает и **автоматически продлевает** TLS-сертификаты для AWS-сервисов (CloudFront, ALB, API Gateway). Снимает главную боль — забытые просроченные сертификаты.

**Межсервисная связь.**
- TLS между всеми внутренними компонентами (не только на периметре). Для микросервисов — **mTLS**, удобно реализуется через **service mesh** (например, App Mesh / sidecar), где шифрование и идентичность сервисов вынесены из кода.
- Внутренние сертификаты — из **ACM Private CA** (приватный удостоверяющий центр).

> **Новые термины — service mesh, ACM Private CA.**
> - **Service mesh** — слой инфраструктуры, который берёт на себя межсервисную связь: mTLS, маршрутизацию, наблюдаемость, не меняя код приложений (через sidecar-прокси).
> - **ACM Private CA (AWS Private Certificate Authority)** — управляемый приватный УЦ для выпуска *внутренних* сертификатов (для mTLS, внутренних доменов), которые не должны быть публично доверенными.

**Контейнерные нагрузки.**
- TLS/mTLS между подами/задачами (mesh или библиотеки), секреты/сертификаты — из Secrets Manager/Private CA, не в образе.
- Шифрование трафика внутри кластера, сетевые политики.

**Обработка больших данных (big data analytics).**
- Шифрование in-transit в EMR/Glue/Redshift/Kinesis (TLS между узлами и при приёме потоков), at-rest — KMS.
- Данные между сервисами ходят по приватным путям (VPC endpoints), не через публичный интернет.

**Операции машинного обучения (ML).**
- **Amazon SageMaker:** шифрование данных в покое (KMS) и трафика обучения/инференса; **VPC-изоляция** обучающих заданий (network isolation), приватные эндпоинты; контроль, чтобы тренировочные данные клиентов не утекали.
- Доступ к данным — по наименьшим привилегиям; аудит обращений.

> **Новый термин — Amazon SageMaker.** Управляемая платформа для ML: подготовка данных, обучение и развёртывание моделей. С точки зрения безопасности важны её опции сетевой изоляции и шифрования, чтобы чувствительные данные клиентов оставались в периметре.

#### TLS: настройка и управление

- Минимум TLS 1.2, предпочтительно 1.3 (быстрее за счёт меньшего числа round-trip — это и про производительность для торговых приложений).
- Через ACM — единая выдача и автопродление публичных сертификатов; через Private CA — внутренних.
- **Security policies** на ALB/CloudFront задают допустимый набор протоколов/шифров централизованно.

#### Жизненный цикл сертификатов

- **Выпуск:** ACM (публичные), Private CA (внутренние).
- **Автопродление:** ACM продлевает сам — устраняет инциденты «упало из-за просроченного сертификата».
- **Мониторинг истечения:** для несmanaged-сертификатов — **ACM + EventBridge/Config**-проверки на приближение даты; алерты заранее.
- **Ротация/отзыв:** при компрометации — отзыв через Private CA (CRL/OCSP) и перевыпуск.

#### Подключение к частной сети

- **VPC Endpoints (PrivateLink)** — доступ к сервисам AWS и собственным сервисам *без выхода в интернет*; трафик идёт по приватной сети AWS. Снижает поверхность атаки и задержку.
- **Transit Gateway** — связь между VPC и регионами по приватным путям.
- Для торговых приложений критична низкая задержка → приватные пути + размещение поближе (cluster placement groups, нужный регион).

> **Новый термин — AWS PrivateLink / VPC Endpoint.** Способ обращаться к сервису (AWS или своему) через приватную точку входа внутри вашей VPC — трафик не выходит в публичный интернет. Это и безопаснее (нет внешней экспозиции), и часто быстрее.

#### Удалённый доступ сотрудников

- **AWS Client VPN** или **AWS Verified Access** для доступа к внутренним ресурсам.
- **Verified Access** — современный zero-trust подход: доступ к приложению предоставляется по проверке *личности и состояния устройства* на каждый запрос, без классического VPN-туннеля.

> **Новые термины — zero trust, AWS Verified Access.**
> - **Zero trust («нулевое доверие»)** — модель, в которой никому не доверяют «по умолчанию» только из-за нахождения внутри сети; каждый доступ проверяется по личности, контексту и состоянию устройства.
> - **AWS Verified Access** — сервис, реализующий zero-trust доступ к внутренним приложениям: проверяет identity (через IdP) и сигнал устройства на каждый запрос, заменяя VPN для веб-приложений.

#### Партнёрская связь

- **mTLS** для API партнёров (взаимная проверка сертификатов).
- **PrivateLink** для прямой приватной связи с партнёром без интернета.
- Cross-account роли с `ExternalId` для контролируемого доступа.

#### Контроль соответствия, ротация ключей, производительность, HA

- **Контроль шифрования:** Config Rules («TLS-only бакеты», «шифрование включено», «нет TLS<1.2»); enforce TLS через bucket/endpoint policy (`aws:SecureTransport`).
- **Ротация ключей:** KMS поддерживает автоматическую годовую ротацию управляемых ключей; ACM сам ротирует сертификаты; mesh-сертификаты — короткоживущие с автообновлением.
- **Влияние на производительность:** TLS 1.3, терминация на edge, session resumption, аппаратное ускорение, приватные пути (PrivateLink) → задержка минимальна; для торговли — особое внимание к размещению и числу хопов.
- **Высокая доступность:** мульти-AZ для эндпоинтов; ACM/KMS — региональные управляемые HA-сервисы; мультирегиональная репликация ключей (multi-Region KMS keys) для согласованного шифрования в разных регионах.

> **Новый термин — multi-Region KMS keys.** Связанные копии одного ключа KMS в нескольких регионах с одинаковым key material — чтобы данные, зашифрованные в одном регионе, можно было расшифровать в другом (нужно для мультирегиональной репликации и DR).

### 5.2. Защита данных в здравоохранении (PHI, неизменяемость, DR)

**Задача.** Глобальная медицинская организация с чувствительными медданными и международными нормами. Требования: PHI шифруется at rest и in transit; медзаписи хранятся 10 лет и неизменяемы; данные исследований реплицируются между регионами; исторические данные автоархивируются по паттернам доступа; в dev — частые бэкапы и восстановление; защита от ransomware; полная аудируемость операций с данными; DR с RPO=1ч и RTO=4ч; оптимизация затрат на долгое хранение; интеграция с он-прем медсистемами. Описать: стратегию шифрования (управление ключами, KMS vs CloudHSM, client-side vs SSE, ротация/доступ); защиту целостности (неизменяемость, версионирование, проверка файлов, цифровые подписи); жизненный цикл данных (retention, переходы классов, оптимизация затрат, соответствие); бэкап/репликацию (механизмы, конфигурации, защита от ransomware, восстановление).

> **Новый термин — PHI (Protected Health Information).** Защищённая медицинская информация — данные о здоровье, идентифицирующие пациента. Регулируется HIPAA (и аналогами). Требует шифрования, аудита, контроля доступа и хранения по правилам.

#### Стратегия шифрования

- **Управление ключами:** **KMS** как основа (envelope encryption во всех сервисах). Для самых строгих требований/единоличного контроля над ключами — **CloudHSM** или **KMS Custom Key Store** поверх CloudHSM.
- **KMS vs CloudHSM:** KMS — проще, дешевле, интегрирован со всем (выбор по умолчанию). CloudHSM — single-tenant, FIPS 140-2 L3, когда регулятор требует, чтобы материал ключа контролировали только вы. Часто гибрид: основное — KMS, особо чувствительное — CloudHSM.
- **Client-side vs SSE:**
  - **SSE (Server-Side Encryption)** — шифрует сам сервис (S3/RDS) на стороне AWS прозрачно (SSE-KMS). Просто и достаточно в большинстве случаев.
  - **Client-side** — данные шифруются *до* отправки в AWS, ключами под вашим контролем; AWS видит только шифротекст. Максимальный контроль для самых чувствительных PHI, ценой сложности.
- **Доступ к ключам и ротация:** строгие **key policies** + grants (наименьшие привилегии: кто может Encrypt/Decrypt); автоматическая ротация; разделение ключей по чувствительности/назначению (отдельные ключи для PHI, логов, бэкапов); полный аудит использования ключей в CloudTrail.

#### Защита целостности данных

- **Неизменяемость:** **S3 Object Lock в режиме Compliance** — объект нельзя удалить/перезаписать до конца срока retention *никому*, включая root. Это и удовлетворяет «10 лет неизменяемо», и защищает от ransomware/инсайдера.
- **Версионирование:** S3 Versioning хранит все версии объекта — случайная/злонамеренная перезапись восстановима.
- **Проверка файлов:** контрольные суммы (checksums) при загрузке/чтении; периодическая сверка целостности.
- **Цифровые подписи:** подпись критичных записей (KMS asymmetric keys / Private CA) для доказательства подлинности и неизменности — важно для медицинско-юридической значимости.

> **Новый термин — S3 Object Lock (WORM).** Write Once Read Many: объект фиксируется на заданный срок. В режиме **Governance** снять блокировку могут особо привилегированные; в режиме **Compliance** — *никто*, даже root, до истечения срока. Главный механизм неизменяемости и анти-ransomware для S3.

#### Управление жизненным циклом данных

- **Retention-политики:** 10 лет с Object Lock; разные сроки для разных категорий.
- **Переходы классов хранения (S3 Lifecycle):** свежее → S3 Standard; реже используемое → Standard-IA; редко → Glacier; архив на годы → Glacier Deep Archive. Либо **S3 Intelligent-Tiering** — сам перемещает объекты между уровнями по фактическим паттернам доступа (отвечает требованию «автоархивирование по паттернам»).
- **Оптимизация затрат:** Deep Archive для многолетнего хранения резко дешевле; Intelligent-Tiering убирает ручное угадывание.
- **Соответствие при этом сохраняется:** Object Lock работает и на архивных классах.

> **Новый термин — S3 Intelligent-Tiering.** Класс хранения, который автоматически перемещает каждый объект между «горячим» и «холодными» уровнями по тому, как часто к нему обращаются — без потери доступности и без ручных правил. Удобно, когда паттерны доступа непредсказуемы.

#### Бэкап и репликация

- **AWS Backup** — централизованное, политиками управляемое резервное копирование множества сервисов (EBS, RDS, DynamoDB, EFS, S3 и др.) с расписаниями и сроками. Для dev — частые бэкапы и быстрое восстановление.
- **Репликация между регионами:** S3 Cross-Region Replication (для данных исследований), мультирегиональные снапшоты, multi-Region KMS keys, чтобы реплики были расшифровываемы.
- **Защита от ransomware (многослойно):**
  - **Object Lock (Compliance)** — нельзя удалить/зашифровать поверх.
  - **Versioning** — откат к доброкачественной версии.
  - **AWS Backup Vault Lock** — неизменяемое хранилище бэкапов (бэкапы нельзя удалить досрочно даже администратору).
  - **Изолированная копия** в отдельном аккаунте/регионе с минимальным доступом (air-gapped-подобно).
  - **MFA Delete** на критичных бакетах.
- **DR с RPO=1ч / RTO=4ч:** репликация и бэкапы с частотой, обеспечивающей точку восстановления ≤1ч; заранее подготовленный (через IaC) и **протестированный** план поднятия в резервном регионе ≤4ч.

> **Новый термин — AWS Backup Vault Lock.** Делает хранилище бэкапов неизменяемым (WORM) на заданный срок: бэкапы нельзя удалить или сократить срок хранения раньше времени — даже root. Ключевой анти-ransomware и compliance-механизм для бэкапов.

#### Аудит и интеграция

- **Аудит всех операций с данными:** CloudTrail (включая data events на S3/Dynamo), логи доступа, использование ключей KMS — всё в неизменяемый Log-аккаунт.
- **Macie** — поиск PHI/PII в S3, контроль, что они лежат корректно и не открыты.
- **Интеграция с он-прем медсистемами:** приватная связь (Direct Connect/VPN), TLS/mTLS, шлюзы (например, Storage Gateway для гибридного хранилища), контроль доступа на стыке.

### 5.3. Криптография, секреты и сертификаты в нескольких регионах

**Задача.** Глобальный банк, расширяющий AWS-инфраструктуру в нескольких регионах. Нужна комплексная криптостратегия и защита данных: управление учётными данными приложений и БД в разных регионах; комбинация импортированных и AWS-сгенерированных ключей; защита чувствительных данных в логах и сообщениях; выпуск и управление сертификатами для внутренних и клиентских приложений; непрерывность при региональных сбоях; соответствие нормам по управлению ключами и защите данных; безопасные межрегиональные операции; автоматическая ротация ключей и сертификатов; защита от утечки и несанкционированного доступа. Описать: управление секретами (хранение/ротация, контроль доступа, мультирегиональная доступность, мониторинг/аудит); управление ключами (импортированные vs AWS-ключи, ротация, external key store, региональная репликация); защиту данных (маскирование, защита логов, безопасность сообщений, обработка чувствительных данных); управление сертификатами (иерархия, региональное распространение, ротация, валидация/мониторинг).

Этот вопрос — «глубокое погружение» в управление секретами/ключами/сертификатами в мультирегионе. Базовые понятия KMS/ACM/Secrets Manager уже введены выше — здесь достраиваем специфику.

#### Управление секретами

- **AWS Secrets Manager** — хранение учётных данных приложений и БД с **автоматической ротацией** (встроенная для RDS и кастомная через Lambda для прочего). Секреты шифруются KMS.
- **Контроль доступа:** resource-политики секрета + IAM по наименьшим привилегиям; разделение, какое приложение к какому секрету.
- **Мультирегиональная доступность:** Secrets Manager умеет **реплицировать секреты в другие регионы** — приложения в каждом регионе читают локальную реплику (меньше задержка, переживает региональный сбой).
- **Мониторинг/аудит:** обращения к секретам — в CloudTrail; алерты на аномальный доступ; GuardDuty.
- **SSM Parameter Store** — для конфигурационных параметров и менее критичных секретов (дешевле; SecureString шифруется KMS).

#### Управление ключами (KMS)

- **Импортированные vs AWS-сгенерированные ключи (BYOK):**
  - AWS-сгенерированные — KMS создаёт и хранит material; проще, поддерживает авторотацию.
  - **Импортированные (BYOK — Bring Your Own Key)** — вы генерируете material сами и импортируете; нужно, когда регулятор/политика требует, чтобы происхождение ключа контролировали вы. Минус: автоматическую ротацию вы организуете сами (импорт нового material).
- **Ротация:** автоматическая годовая для AWS-управляемого material; для импортированных — процедурная ротация по расписанию.
- **External Key Store (XKS):** KMS может использовать ключи, физически хранящиеся *вне* AWS (в вашем HSM), — для самых строгих требований «ключи только у нас». AWS при этом не имеет доступа к самому material.
- **Региональная репликация:** **multi-Region keys** для согласованного шифрования/дешифрования в разных регионах и для DR.

> **Новые термины — BYOK и External Key Store (XKS).**
> - **BYOK** — импорт собственного материала ключа в KMS, чтобы контролировать его происхождение.
> - **XKS (External Key Store)** — режим, где сам ключевой материал живёт в вашем внешнем HSM, а KMS лишь оркестрирует операции; AWS физически не владеет ключом. Применяют под жёсткие требования суверенитета ключей.

#### Защита данных (маскирование, логи, сообщения)

- **Маскирование данных:**
  - В логах — не писать чувствительное вовсе; маскировать на уровне приложения; **CloudWatch Logs data protection** автоматически обнаруживает и маскирует чувствительные паттерны (карты, SSN) в логах.
  - В аналитике/доступе — динамическое маскирование (например, маскирование колонок в Redshift/Lake Formation по правам).
- **Защита логов:** шифрование KMS, неизменяемость, ограниченный доступ, обнаружение секретов в логах.
- **Безопасность сообщений:** очереди/шины (SQS/SNS/Kinesis) — шифрование KMS at rest + TLS in transit; для особо чувствительного — шифрование payload на стороне клиента до помещения в очередь.
- **Обработка чувствительных данных:** Macie для обнаружения; токенизация/шифрование на уровне поля для самого критичного (номера счетов).

> **Новый термин — CloudWatch Logs data protection.** Функция, которая сканирует логи на чувствительные данные по управляемым и кастомным детекторам и автоматически их маскирует, оставляя аудит-след о факте обнаружения. Прямо отвечает требованию «защитить чувствительные данные в логах».

> **Новый термин — токенизация.** Замена чувствительного значения (номер карты) на несекретный «токен», по которому исходное значение хранится отдельно в защищённом хранилище. Снижает PCI-scope: системы работают с токенами, а не с реальными номерами.

#### Управление сертификатами

- **Иерархия:** корневой УЦ (root CA, оффлайн, максимально защищён) → промежуточные (intermediate) CA → конечные сертификаты. Реализуется через **ACM Private CA**. Публичные сертификаты для клиентских приложений — через ACM (публичный).
- **Региональное распространение:** Private CA и ACM — региональные; выстраивают выпуск/доверие в каждом нужном регионе; внутренние корневые/промежуточные распространяются на нагрузки.
- **Ротация:** ACM автопродляет публичные; внутренние — короткоживущие с автообновлением (mesh/Private CA); процедуры отзыва (CRL/OCSP).
- **Валидация и мониторинг:** проверка цепочки доверия; мониторинг сроков (EventBridge/Config-алерты на приближение истечения); алерты на неожиданный выпуск (контроль за тем, кто что выпустил).

#### Непрерывность при региональных сбоях

- Multi-Region KMS keys + репликация секретов + Private CA в нескольких регионах → приложения переживают отказ региона без потери способности шифровать/расшифровывать и аутентифицироваться.
- Заранее протестированный DR-переключатель.

#### Соответствие и защита от утечек

- Управление ключами по нормам (KMS/CloudHSM/XKS под нужный уровень контроля), полный аудит ключевых операций.
- Наименьшие привилегии к ключам/секретам, MFA на чувствительные действия.
- Обнаружение утечек: Macie, GuardDuty, Access Analyzer, CloudWatch Logs data protection.

## Раздел 6. Governance и соответствие требованиям

> **Что такое governance в облаке.** Это набор правил, структур и автоматизмов, которые гарантируют, что *все* аккаунты и команды работают единообразно и безопасно по умолчанию — даже когда их сотни и они растут. Без governance каждая команда «изобретает» свою безопасность, и появляются дыры. Главные инструменты: AWS Organizations, Control Tower, SCP, и автоматизация политик.

### 6.1. Multi-account governance после поглощений

**Задача.** Глобальная компания быстро растёт в AWS, поглощает другие компании и должна интегрировать их AWS-аккаунты, сохраняя строгий контроль. Нужны: хорошо структурированная мультиаккаунт-среда; соответствие международным нормам; стандартные базовые настройки безопасности для всех аккаунтов; централизованный мониторинг и управление; защита root; автоматизированные guardrails; чёткие процедуры экстренного доступа; интеграция существующих аккаунтов; масштабируемость контролей. Описать: архитектуру Organizations (структура аккаунтов, дизайн OU, реализация политик, стратегию интеграции поглощённых); систему контролей (Control Tower, обязательные/необязательные guardrails, кастомные, мониторинг соответствия); управление правами (SCP, RAM, AI opt-out, tag policies); централизованную безопасность (делегированный админ, конфигурация security-сервисов, кросс-аккаунт мониторинг, авто-ответы); защиту root (управление кредами, MFA, break-glass, мониторинг доступа).

#### Архитектура AWS Organizations

**Структура и иерархия аккаунтов.** Базовый паттерн (AWS multi-account best practice / landing zone):
- **Management account (он же payer)** — корень организации; в нём *ничего не запускают*, кроме управления организацией. Максимально защищён.
- **Security/Audit account** — делегированный администратор GuardDuty/Security Hub/Config; сюда стекаются находки.
- **Log Archive account** — централизованные неизменяемые логи.
- **Workload-аккаунты** — отдельно прод, dev, test, по бизнес-юнитам/продуктам.
- **Shared Services / Network account** — общая сеть, общие сервисы.

**Дизайн OU (Organizational Units).** Аккаунты группируют в **OU** — папки, к которым применяют общие политики:
- `Security` OU (Security, Log Archive),
- `Infrastructure` OU (Network, Shared Services),
- `Workloads` OU → `Prod` и `Non-Prod` под-OU,
- `Sandbox` OU (песочницы с жёсткими лимитами),
- `Suspended` OU (карантин для аккаунтов на расследовании/выводе).

> **Новый термин — OU (Organizational Unit).** Контейнер для группировки аккаунтов в Organizations. К OU применяются SCP и другие политики, которые наследуются всеми вложенными аккаунтами. Это позволяет управлять сотнями аккаунтов через десяток OU.

> **Новый термин — landing zone.** Заранее спроектированный, безопасный «каркас» мультиаккаунт-среды (структура OU, базовые аккаунты, guardrails, логирование), на который «приземляют» рабочие нагрузки. AWS Control Tower автоматизирует его создание.

**Реализация политик** — через SCP (границы), tag policies (стандарт тегов), backup policies, AI services opt-out (ниже).

**Стратегия интеграции поглощённых компаний.** Это центр вопроса. Порядок:
1. **Инвентаризация и оценка** чужого аккаунта (что там, какие риски, какие открытые дыры) — Security Hub/Config/Inspector + ручной аудит.
2. **Приглашение аккаунта в Organizations** (invite existing account) и помещение его сначала в **карантинный/Onboarding OU** с ограничительными SCP.
3. **Приведение к baseline:** включить логирование, GuardDuty, Config, базовые guardrails; навести порядок в IAM (убрать долгоживущие ключи, лишних админов, защитить root).
4. **Постепенный перенос** в нужное «постоянное» OU после соответствия baseline.
5. Где проще — миграция нагрузок в новые «чистые» аккаунты, а старый выводится.

#### Система контролей: AWS Control Tower

**AWS Control Tower** — сервис, который автоматически разворачивает и поддерживает безопасную landing zone: создаёт базовые аккаунты, OU, включает логирование и набор **guardrails (controls)**.
- **Обязательные (mandatory) controls** — всегда включены (например, запрет менять конфигурацию логирования Control Tower).
- **Strongly recommended / elective controls** — включаете по необходимости (запрет публичных бакетов, требование шифрования EBS и т.п.).
- Контроли бывают **превентивные** (через SCP — *не дать сделать*), **детективные** (через Config — *заметить нарушение*) и **проактивные** (через CloudFormation hooks — *поймать на этапе развёртывания, до создания ресурса*).
- **Account Factory** — стандартизированное создание новых аккаунтов уже с baseline.
- **Кастомные controls** — свои SCP/Config-правила под специфические нормы.
- **Мониторинг соответствия** — дашборд Control Tower + Security Hub стандарты.

> **Новый термин — guardrail (control).** Правило-ограничитель, применяемое ко всем аккаунтам OU. Превентивный guardrail запрещает действие (SCP), детективный — обнаруживает нарушение (Config), проактивный — блокирует несоответствующий ресурс на этапе деплоя. Это способ задать «безопасно по умолчанию» в масштабе.

#### Управление правами на уровне организации

- **SCP (Service Control Policies):** задают максимально допустимые действия. Примеры: запретить отключать CloudTrail/GuardDuty; запретить регионы вне разрешённых; запретить создание публичных ресурсов; защитить root-действия.
- **AWS RAM (Resource Access Manager):** безопасное **совместное использование ресурсов** между аккаунтами (подсети, Transit Gateway, Private CA) без копирования и без широких прав. (Подробнее — в 6.2.)
- **AI services opt-out policy:** политика организации, запрещающая AWS использовать ваши данные из AI-сервисов для улучшения моделей — важно для конфиденциальности/соответствия.
- **Tag policies:** принуждают единый стандарт тегирования (об этом — 6.2), что критично для распределения затрат, ABAC и инвентаризации.

> **Новый термин — AWS RAM (Resource Access Manager).** Сервис для контролируемого шеринга конкретных ресурсов между аккаунтами одной организации — вместо дублирования инфраструктуры. Например, одна сеть в Network-аккаунте, расшаренная на workload-аккаунты.

#### Централизованная безопасность

- **Делегированный администратор** для GuardDuty/Security Hub/Config/Inspector/Detective → всё управление и все находки в Security-аккаунте, авто-подключение новых аккаунтов.
- **Кросс-аккаунт мониторинг** — единое окно по всей организации.
- **Авто-ответы** — EventBridge → Lambda/Step Functions на находки (как в разделах 1–2), централизованно.

#### Защита root-аккаунта

Root — всемогущий владелец аккаунта, главная цель атаки. Меры:
- **Управление кредами:** убрать root access keys полностью; сложный уникальный пароль в защищённом хранилище; root не используют для повседневных задач.
- **MFA на root** — обязательно, желательно аппаратный ключ; для management-аккаунта — особо строго.
- **Централизованное управление root** (новая возможность Organizations) — позволяет на уровне организации убрать/заблокировать root-креды у member-аккаунтов, резко снижая поверхность атаки.
- **Break-glass процедуры** — доступ к root только по аварийной, задокументированной процедуре, с MFA, двумя людьми (dual control) и громким аудитом.
- **Мониторинг доступа к root** — CloudWatch alarm + EventBridge на *любой* вход root или его действие → немедленный алерт SOC.

#### Соответствие и масштабируемость

- Международные нормы (HIPAA/GDPR/PCI/локальные) → соответствующие OU/контроли, резидентность данных через ограничение регионов SCP, неизменяемые логи, отчёты Security Hub/Audit Manager.
- Масштабируемость: Control Tower Account Factory + guardrails на OU + делегированные админы → каждый новый (в т.ч. поглощённый) аккаунт автоматически получает baseline.

### 6.2. Безопасное и единообразное развёртывание (IaC + sharing)

**Задача.** Глобальная компания с cloud-first стратегией в нескольких аккаунтах. Требования: единообразное безопасное развёртывание инфраструктуры во всех аккаунтах; стандартные контроли и настройки безопасности; распределение затрат и управление ресурсами по бизнес-юнитам; безопасный шеринг ресурсов между dev и prod; соответствие нормам; автоматизированное принуждение политик; масштабируемость; учёт и управление ресурсами. Описать: подход к IaC (безопасность и валидация шаблонов, кросс-аккаунт развёртывание, проверка соответствия, версионирование и управление изменениями); организацию ресурсов (стандарты тегов и их принуждение, группировку ресурсов, распределение затрат, контроль доступа); централизованное управление политиками (развёртывание, принуждение конфигураций, мониторинг, авто-remediation); шеринг ресурсов (механизмы доверенного обмена, контроль доступа, мониторинг использования, соответствие).

#### Подход к IaC (Infrastructure as Code)

**Инструменты:** **AWS CloudFormation** (родной), **Terraform** (мультиоблачный), **AWS CDK** (инфраструктура на обычных языках программирования поверх CloudFormation).

**Безопасность и валидация шаблонов** (то самое «shift-left» для инфраструктуры):
- **Сканирование шаблонов** до деплоя: **cfn-guard**, **checkov**, **tfsec**, **Trivy** — ловят небезопасные конфигурации (открытый бакет, security group `0.0.0.0/0`, отсутствие шифрования) ещё в pull request.
- **CloudFormation Guard / Hooks** — политика-как-код, блокирующая несоответствующие ресурсы на этапе создания (проактивный контроль).
- **Линтинг и валидация** синтаксиса; запрет «магических» ручных правок в обход кода.

**Кросс-аккаунт развёртывание:**
- **CloudFormation StackSets** — разворачивают один шаблон сразу во множество аккаунтов и регионов из одного места (идеально для baseline-безопасности на всю организацию, с авто-применением к новым аккаунтам).
- Для Terraform — пайплайны с assume role в целевые аккаунты (через OIDC, без статических ключей).

**Проверка соответствия:** в пайплайн встроены policy-as-code проверки; после деплоя — Config Rules/Security Hub подтверждают состояние.

**Версионирование и управление изменениями:** всё в Git; изменения только через pull request с обязательным ревью и автоматическими проверками; история = аудиторский след; разделение прав «кто пишет» и «кто аппрувит прод».

> **Новые термины — StackSets, policy-as-code.**
> - **CloudFormation StackSets** — механизм развёртывания одного набора ресурсов во многих аккаунтах/регионах централизованно; поддерживает автоматическое применение к новым аккаунтам OU.
> - **Policy-as-code** — правила безопасности/соответствия, записанные как код (cfn-guard, OPA/Conftest, checkov), которые автоматически проверяют инфраструктуру в пайплайне. Убирает ручные ревью-чек-листы.

#### Организация ресурсов

- **Стандарты тегов и принуждение:** обязательные теги (`Environment`, `Owner`, `CostCenter`, `DataClassification`, `Project`). Принуждение через **Tag Policies** (организация) + Config-правила + проактивные хуки (нет тега — нет деплоя).
- **Группировка ресурсов:** **AWS Resource Groups** + теги для логического объединения по проекту/юниту.
- **Распределение затрат:** **Cost Allocation Tags** + **AWS Cost Explorer / Budgets** → видно, сколько тратит каждый бизнес-юнит; алерты на превышение бюджета.
- **Контроль доступа:** **ABAC по тегам** (раздел 4) — доступ к ресурсам своего проекта/юнита по тегам, что масштабируется без взрыва числа ролей.

> **Новый термин — Cost Allocation Tags.** Теги, которые AWS учитывает в биллинге, позволяя разрезать затраты по проектам/командам/средам. Основа честного распределения расходов в мультиаккаунт-среде.

#### Централизованное управление политиками

- **Развёртывание политик безопасности:** baseline (логирование, guardrails, security-сервисы) раскатывается StackSets-ами и Control Tower на все аккаунты.
- **Принуждение конфигураций:** SCP (превентивно) + Config Rules (детективно) + Hooks (проактивно).
- **Мониторинг соответствия:** Security Hub стандарты + Config conformance packs (наборы правил под стандарт) + дашборды.
- **Авто-remediation:** Config + EventBridge → SSM Automation/Lambda автоматически чинят дрейф (закрыть публичный доступ, включить шифрование) или поднимают тикет.

> **Новый термин — Config conformance pack.** Готовый набор Config-правил и действий remediation, упакованный под конкретный стандарт (например, «операционные best practices для PCI DSS»), который разворачивается на аккаунты разом.

#### Шеринг ресурсов (dev ↔ prod и между юнитами)

- **AWS RAM** — доверенный шеринг конкретных ресурсов (подсети, Transit Gateway, Private CA, Resolver rules) между аккаунтами без дублирования и без широких прав.
- **Контроль доступа при шеринге:** делиться по наименьшим привилегиям, только нужными ресурсами, только нужным аккаунтам/OU; разделять dev и prod (prod-данные не шарят в dev).
- **Lake Formation** — для шеринга данных озера с гранулярными правами (таблицы/колонки/строки).
- **Мониторинг использования:** CloudTrail + Cost Explorer по тегам + Config — кто чем шаренным пользуется.
- **Соответствие:** шеринг чувствительных данных — под классификацией (тег `DataClassification`) и контролем; аудит обращений.

#### Масштабируемость

StackSets + Control Tower + теги + ABAC + RAM = новый аккаунт/юнит подключается по шаблону, получает baseline, корректное распределение затрат и контролируемый доступ к общим ресурсам автоматически.

### 6.3. Непрерывный мониторинг соответствия (HIPAA/GDPR/PCI DSS)

**Задача.** Глобальная медицинская организация на AWS, обязанная соответствовать HIPAA, GDPR, PCI DSS и демонстрировать непрерывный мониторинг и отчётность. Нужно: отслеживать соответствие на сотнях ресурсов; формировать доказательную базу для регулярных аудитов; оценивать архитектуру на best practices; быстро реагировать на нарушения; вести детальные аудит-журналы; автоматизировать соответствие где возможно; демонстрировать непрерывный мониторинг; формировать исчерпывающие отчёты. Описать: мониторинг соответствия (настройка правил, уведомления, авто-remediation, сбор/управление алертами); сбор аудиторских доказательств (методы, организация/категоризация, retention, контроль доступа); оценку архитектуры (Well-Architected, оценка best practices, gap-анализ, отслеживание улучшений); отчётность/документацию (отчёты о соответствии, аудит-журнал, управление доказательствами, коммуникация со стейкхолдерами).

> **Пояснение норм.** **HIPAA** — защита медданных (PHI) в США. **GDPR** — защита персональных данных в ЕС (права субъектов, согласие, уведомление об утечках в 72ч, ограничения трансграничной передачи). **PCI DSS** — стандарт защиты данных платёжных карт. Они пересекаются по контролям (шифрование, доступ, аудит), но имеют и специфику. Цель — *непрерывное* доказуемое соответствие, а не разовая подготовка к аудиту.

#### Мониторинг соответствия

- **AWS Config + Conformance Packs** — основа: правила «как должно быть» под каждый стандарт, постоянно проверяющие сотни ресурсов и записывающие историю состояний.
- **AWS Security Hub** — встроенные стандарты (PCI DSS, AWS FSBP, CIS) с оценкой в процентах; единая очередь нарушений.
- **Уведомления:** EventBridge → SNS на нарушения; маршрутизация по серьёзности (критичное — дежурным, остальное — тикеты).
- **Авто-remediation:** Config + SSM Automation/Lambda — авто-исправление типовых нарушений (включить шифрование, закрыть публичный доступ) с аудитом каждого действия.
- **Сбор и управление алертами:** агрегация в Security Hub; подавление известных исключений; приоритизация.

#### Сбор аудиторских доказательств

- **AWS Audit Manager** — ключевой сервис: автоматически **собирает доказательства** (из Config, CloudTrail, Security Hub и др.) и маппит их на контроли готовых фреймворков (HIPAA, GDPR, PCI DSS, SOC 2). Превращает «непрерывный мониторинг» в готовые наборы доказательств к аудиту.
- **Методы сбора:** автоматический (Audit Manager) + ручные артефакты (политики, процедуры) при необходимости.
- **Организация/категоризация:** по фреймворку и контролю (Audit Manager делает это сам).
- **Retention:** доказательства и логи — в неизменяемое хранилище (S3 + Object Lock) на требуемый срок.
- **Контроль доступа:** к доказательствам — наименьшие привилегии, аудит обращений.

> **Новый термин — AWS Audit Manager.** Сервис, который непрерывно собирает доказательства соответствия из ваших AWS-данных и раскладывает их по контролям выбранного стандарта, генерируя готовые к аудиту наборы. Резко снижает ручной труд подготовки к проверкам.

#### Оценка архитектуры

- **AWS Well-Architected Framework** — методология оценки по 6 столпам (операционное превосходство, **безопасность**, надёжность, эффективность, оптимизация затрат, устойчивость). **Well-Architected Tool** проводит структурированный ревью, выявляет риски.
- **Оценка best practices:** Security Hub (FSBP/CIS) + Trusted Advisor (проверки безопасности/лимитов).
- **Gap-анализ:** разница между текущим состоянием и требованиями стандарта/Well-Architected → список улучшений.
- **Отслеживание улучшений:** план исправлений с приоритетами и сроками; повторные прогоны показывают прогресс (тренд оценки соответствия во времени).

> **Новые термины — Well-Architected Framework / Tool, Trusted Advisor.**
> - **Well-Architected** — свод best practices AWS и инструмент самооценки по ним; столп Security особенно релевантен.
> - **Trusted Advisor** — сервис автоматических проверок аккаунта (безопасность, отказоустойчивость, затраты, лимиты) с рекомендациями.

#### Отчётность и документация

- **Отчёты о соответствии:** из Audit Manager (наборы доказательств по контролям), Security Hub (постура и тренды), Config (история соответствия). Для дашбордов руководству — QuickSight поверх выгрузок.
- **Ведение аудит-журнала:** CloudTrail (вкл. data events) в неизменяемый Log-аккаунт — кто что делал с ресурсами и данными.
- **Управление доказательствами:** централизованно в Audit Manager + неизменяемый S3; версионирование; retention под каждый стандарт.
- **Коммуникация со стейкхолдерами:** регулярные отчёты о постуре безопасности и соответствии (тренды, открытые риски, план), разные срезы для технических команд, руководства и аудиторов.
- **AWS Artifact** — портал, где берут готовые отчёты о соответствии *самой AWS* (SOC, ISO, PCI AOC) — доказательство соответствия инфраструктурного провайдера, нужное вашим аудиторам.

> **Новый термин — AWS Artifact.** Самостоятельный портал с документами о соответствии и сертификатами AWS как провайдера (и с управлением соглашениями вроде BAA). Это доказательная база «нижнего слоя» (что сама платформа сертифицирована), которую вы дополняете доказательствами своего слоя из Audit Manager.

#### Как всё связывается в «непрерывное соответствие»

```
Config Rules / Conformance Packs ─┐
Security Hub стандарты ────────────┼─► непрерывная оценка сотен ресурсов
GuardDuty / Inspector ─────────────┘
            │
            ├─► нарушение → EventBridge → авто-remediation + алерт
            │
            ▼
   AWS Audit Manager ──► сбор доказательств по HIPAA/GDPR/PCI
            │
            ▼
   Неизменяемое хранилище (S3 Object Lock) + отчёты (QuickSight/Artifact)
            │
            ▼
   Аудиторы / руководство / регуляторы
```

Главная идея зрелого подхода: соответствие — это **не проект «к аудиту», а постоянно работающий конвейер**, где состояние проверяется автоматически, нарушения чинятся сразу, а доказательства копятся непрерывно.

## Сквозные принципы (шпаргалка)

Эти идеи повторялись во всех разделах — вот они одним списком, чтобы держать в голове.

1. **Наименьшие привилегии (least privilege).** Всегда выдавать минимум прав; расширять по фактической потребности (Access Analyzer помогает урезать).
2. **Временный доступ вместо постоянных ключей.** Роли + STS + федерация; долгоживущие access keys — антипаттерн.
3. **Защита в глубину (defense in depth).** Несколько слоёв (сеть, identity, шифрование, мониторинг), чтобы пробитие одного не открывало всё.
4. **Минимизация blast radius.** Изоляция через отдельные аккаунты, сегментация сети, отдельные ключи.
5. **Безопасно по умолчанию (secure by default).** Guardrails (SCP/Control Tower), baseline через StackSets — новый аккаунт уже безопасен.
6. **Всё как код.** IaC + policy-as-code + «мониторинг как код» → повторяемость, ревью, аудит.
7. **Централизованная наблюдаемость.** Логи и находки стекаются в выделенные Security/Log аккаунты; единое окно (Security Hub).
8. **Шифрование везде.** At rest (KMS) и in transit (TLS), управление ключами под нужный уровень контроля.
9. **Неизменяемость критичного.** Object Lock / Backup Vault Lock / неизменяемые логи — против ransomware и подделки.
10. **Автоматизация реакции.** EventBridge → Lambda/Step Functions/SSM; разрушительное — с подтверждением человека.
11. **Непрерывное соответствие.** Config + Security Hub + Audit Manager вместо разовой подготовки к аудиту.
12. **Тестировать то, на что рассчитываешь.** Game days, проверка DR (RTO/RPO), WAF-правила в режиме count перед block, canary-проверки мониторинга.
13. **Защита root и break-glass.** Root под MFA и почти не используется; аварийный доступ — задокументирован и громко аудируется.
